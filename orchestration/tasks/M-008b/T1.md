The existing progress-aware deadline may still extend on body-text growth, but it must be **capped** so the method can never wait past `start + ceiling`. When the absolute ceiling is reached with no completion, **fail closed** with `ResponseTruncatedError` (already imported; same error the timeout path raises).

- New module constant, clearly named, e.g. `_REAL_COMPLETION_CEILING_S = 600.0` (10 min). **Justify the default in your report**: it must be generously larger than any legitimate single-turn ChatGPT stream (even long deep-research/code turns finish within a few minutes of active streaming), so it never clips a real response, while bounding a pathologically growing/oscillating body. - New OPTIONAL parameter on the method so tests can inject a small ceiling: `def wait_for_completion(self, timeout_s: float = 120.0, max_total_wait_s: float | None = None)`. `None` ⇒ use `_REAL_COMPLETION_CEILING_S`. Keep it configurable; do not hardcode the ceiling inside the loop.
