## Checklist — for EACH item: quote the design's handling, mark `HONORED | PARTIAL | VIOLATED | MISSING`, and give the evidence
1. **Auth/OAI headers.** Capture endpoint `GET /backend-api/conversation/<id>`; cookies/accept-only → **404**; works only with the web-app headers (`authorization` bearer + `oai-client-build-number`/`-version` + `oai-device-id` + `oai-language` + `oai-session-id` + `x-openai-target-path` + `x-openai-target-route`). Does the design **obtain them from the page's own request** and **never persist/log the token**? Does it **fail-closed** if they can't be obtained?
2. **Scale / streaming.** ~17.1 MB, ~5k mapping nodes, **no pagination**. Does the design handle this with streaming/efficient handling (not a gratuitous full-in-memory parse), and is the 17MB figure used as **measured** (not re-invented)?
3. **Content extraction.** Visible assistant text in `content.parts` (list of strings); `assistant:code` → `content.text`; `assistant:thoughts` → `content.thoughts`; many tool/code/thought nodes have no `parts`. Is the per-node extraction rule correct and complete?
4. **DR/Pro turns.** No `content_type=="deep_research"`; DR = large `turn_exchange_id` groups (one user msg + hidden `thoughts`/`code`/`reasoning_recap` + tool nodes `web.run`/`file_search`/`python`/`container.exec` + one visible final `assistant:text` report in `content.parts[0]`). Citations in `metadata.content_references`/`citations`/`search_result_groups`/`search_queries`. Does the design group + linearize + capture citations correctly, with NO bespoke DR special-casing beyond grouping?
5. **Attachment shapes.** All four: `metadata.attachments[]`; `content_references[type=file]`; `content.assets[].asset_pointer`; `metadata.aggregate_result`. No literal `/files/` URLs (ids/pointers only). Does the schema represent all four? Are citations kept separate from attachments?
6. **Completion signals.** Top-level `async_status`; metadata `is_complete`/`is_finalizing`/`pro_progress`; node `status`; possible `/stream_status`. Does completion detection use these (and treat `/stream_status` as a hypothesis to confirm in M5)?
7. **Selectors.** composer `#prompt-textarea` (contenteditable, no `data-testid=prompt-textarea`); tools `button[data-testid="composer-plus-btn"]`; copy `button[data-testid="copy-turn-action-button"]`; turns `[data-message-id][data-message-author-role]`; **model picker label-driven (NO stable test-id)**; menus are Radix portals (`[data-radix-popper-content-wrapper]`, tiers `menuitemradio`, families `menuitem` submenu). Clipboard `readText` exists but permission=`prompt` (grant-only → not an unattended fallback). Are all these encoded correctly?
8. **Projects.** Deferred/not-live-verified in M2 (`gizmo_id`/`gizmo_type` null on `/c/<id>`). Does the design treat project send/create as a near-term assumption to confirm in M5 rather than claiming it's verified?
