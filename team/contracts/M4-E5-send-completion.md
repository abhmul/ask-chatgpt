## Behaviors (map E1–E12; gotchas #2 and #3 are the heart)
- **E6 [gotcha #2]:** `verify_prompt_submitted` succeeds ONLY when a user turn newer than baseline (new latest user id OR increased user count) carries the normalized prompt; a no-op submit (user turns unchanged), or a new user turn with WRONG text, raises `PromptNotSubmittedError` and `wait_for_completion` is NEVER called and NO stale assistant is returned.
- **E3:** read baseline (latest user/assistant ids + counts) BEFORE any composer mutation; idle-reload first; pass the pre-send assistant baseline UNCHANGED to completion; `query_turns` precedes fill/click (assert via the mock call log).
- **E5:** `wait_for_composer` tolerates transient absent→absent→visible until timeout (fake clock); permanent absence → `SelectorNotFoundError`, no completion; fill verifies normalized composer text; submit clicks an ENABLED send control after input or Enter fallback only while composer focused (never blind global Enter; never click a disabled button and report success).
- **E4:** `Session.ask` calls `Store.begin_send` BEFORE composer/submit can fail; on composer-never-appears / no-op submit, the pending stub is hidden from default reads but present with `include_pending=True` (lose-nothing).
- **E7:** on success, `commit_send` supersedes the stub and `ask` returns the NEW assistant `TurnRecord` (newer id, `user_message_id` linking the submitted user) — never the user record, the stub, a `CompletionState`, or a pre-existing assistant.
- **E8:** `wait_for_completion` invoked only after `verify_prompt_submitted`; success requires an assistant id different from AND newer than `baseline.latest_assistant_id` (old stable assistant is NOT completion).
- **E9:** conservative completion — a new id alone is insufficient while text empty (without explicit empty-complete) or while any relevant `async_status`/node status/`is_complete=False`/`is_finalizing`/`pro_progress` is active/unknown-active; DOM consensus requires new id + stop-absent for a stable window + text hash/length stable that window + `(saw_streaming or non_empty)`.
- **E10 [gotcha #3]:** `activity_timeout_s` measures time since LAST progress (not wait start); progress resets on `update_time`/`current_node`/new node id/new assistant id/text hash/text length/`async_status`/node status/`pro_progress`/`is_finalizing` (same-length different-text counts); `max_total_wait_s=None` runs PAST 600s under continuous fake-clock progress (a hidden 600s ceiling MUST make this test fail); explicit `max_total_wait_s` exceeded → `MaxTotalWaitExceededError`; no-activity window elapsed → `CompletionTimeoutError` WITH partial salvage persisted.
- **E11:** with `progress_poll_interval_s=2`, `backend_check_interval_s=30` (fake clock), backend checks are strictly sparser than DOM polls; headers are one-use (mock flags reuse); no full ~17MB-equivalent fetch per DOM tick (assert via mock counters).
- **E12 [DECISION 13/Q6]:** salvage order backend-partial → copy(only with explicit grant) → DOM; default clipboard `prompt` → `HumanActionNeededError`; persisted partial has honest `status`(`partial`/`error`)/`capture_source`/`fidelity` + redacted error (no header/prompt canaries); never returns partial as a complete success.
- **E2:** requested `--model`/`--tool` change in M4 → menus stub fails closed (`ModelSelectionNotReflectedError`/`ToolSelectionNotReflectedError`) before submit; no canonical user committed; a default read shows no confirmed user.
