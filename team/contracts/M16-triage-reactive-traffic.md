## LENS-3 — Reactive-completion feasibility (the load-bearing technical risk) → `team/evidence/handoffs/M16/lens3-reactive-completion.md`
**Goal:** Determine, from code, whether "the send's response stream closing ⇒ turn is done" is observable via the EXISTING CDP seam — the single biggest technical risk in the whole redesign.
- Trace the SEND path: `send.py:submit_composer` (~`:178`) → `send_prompt` (~`:259`) → `verify_prompt_submitted` (~`:212`). When the composer submits, the SPA opens a streaming/SSE response from `/backend-api/...`; identify which request that is and how the page delivers chunks.
- Examine the existing observation seam: `cdp.py:wait_for_request` (`:754`, header-aware Network observation) and the binding-stream `StreamState`/`consume_stream_event`/`{kind:"done"}` (`:78,449,491,509,817`). **Decide whether these can observe the SPA's OWN native send-response stream, or whether they only stream the tool's OWN `fetch()` (capture) calls.** This distinction is decisive: if the binding-stream is tool-fetch-only, reactive completion needs a NEW mechanism (CDP `Network.responseReceived`/`loadingFinished`/`EventSourceMessageReceived` on the SPA's stream), not just reuse.
- Identify exactly what must be TRUE on a real attended leg to confirm stream-close == completion (e.g. "Network `loadingFinished` fires on the conversation stream request when and only when the turn ends"), and **design that minimal probe** (CDP domains to enable, events to capture, the single short send that would confirm it) — DESIGN ONLY, do not run.
- Cross-check against the team lesson that completion is subtle (short multi-turn responses clip; current heuristic = streaming-seen + stop-absence ≥3 s + text-stable ≥3 s + global-marker fallback): would a pure stream-close signal be MORE or LESS robust than today's heuristic, and should the redesign keep a fallback?
- **Deliverable:** verdict `OBSERVABLE-VIA-EXISTING-SEAM` / `NEEDS-NEW-NETWORK-OBSERVER` / `UNCERTAIN-NEEDS-PROBE`, the exact mechanism + file:line evidence, the minimal confirming probe spec, and a recommendation on keeping a fallback.

## LENS-5 — Rate-governor + 429 detection + shared-resource-ceiling design → `team/evidence/handoffs/M16/lens5-rate-governor.md`
**Goal:** Design the request-rate governor (the deferred `issues/2026-06-21-*` feature, now in-scope as Layer C-4) and the 429 fail-soft path, grounded in the shared-resource-ceiling rigor.

Concurrent consumers (the driver + the operator's own browsing + possibly another agent) do not coordinate via independent per-process limiters. How is a human-plausible ceiling derived (it is operator-owned; state that the numeric value must be confirmed with the operator, not guessed — memory `verify-inherited-resource-claims`)?
- **Cross-process home (the hard part, M12):** today each `ask`/`scrape` is a fresh process; `AdaptiveSendBudget` is in-memory, send-only, single-process, `record_soft_signal` unwired (verify in code). A governor that throttles ALL requests under a global cap needs cross-process state (a file-lock token bucket? a long-lived `loop` session that owns the budget?). Design the minimal mechanism and say where it lives.
- **429 / "Too many requests" detection → fail-soft:** specify detecting the 429 status / the modal, honoring `Retry-After`, emitting a DISTINCT exit code, and backing off — instead of the current path where backend-capture INTERNAL_ERROR fell back to clipboard → `clipboard fallback requires explicit permission` → exit 21 (the incident). Where in `capture.py`/`completion.py`/`cli.py`/`errors.py` does this slot in?
- **Human pacing + jitter:** extend `_sleep_until_spacing_allows_submit` (`session.py:223`) / `AdaptiveSendBudget` with a jittered politeness delay between sends.
- **Deliverable:** the governor design (mechanism, location, cross-process state), the budget-allocation formula with the "ceiling is operator-owned, confirm it" caveat, the 429 fail-soft spec mapped to file:line, and the pacing/jitter change.
