2. **Write + run a probe script** with `uv run python <script under the scratchpad or tmp/>` (project venv — has playwright + can `import` the tool's `cdp.py` for reference). The script, mirroring `src/ask_chatgpt/channels/cdp.py:595-599,635-639,892-902`:
   - `connect_over_cdp("http://127.0.0.1:9222", timeout=5000)`; take the existing default context; **open your OWN page** `context.new_page()` (this is your only tab — never touch others).
   - `session = context.new_cdp_session(page)`; `session.send("Network.enable")`; register listeners for `Network.requestWillBeSent`, `Network.responseReceived`, `Network.loadingFinished`, `Network.loadingFailed`. For EACH event record ONLY `{eventName, requestId, method, path (host+query+ids STRIPPED), resourceType, status, monotonic_ts}` into an in-memory list. NEVER bodies/headers/cookies/content.
   - `page.goto("https://chatgpt.com/?temporary-chat=true")`; wait for load. **Confirm you are on a Temporary Chat** (the temp-chat UI indicator / URL) and the composer exists. If you hit a login page / Cloudflare "Just a moment" / no composer → `BLOCKED:<reason>`, close your page, stop. If a "Too many requests"/rate modal ever appears → `BLOCKED:rate_limit`, stop.
   - Type the throwaway prompt `Reply with exactly: pong` into the composer; record `t0_send` (monotonic) immediately before clicking send; click send ONCE.
   - Observe until turn-end or a 120 s cap: identify the assistant send-response **stream** request — the `/backend-api/*` request that starts after `t0_send` and streams the answer (likely a POST to a conversation endpoint, `resourceType` fetch/eventsource, long-lived). Record when its `Network.loadingFinished` (or `loadingFailed`) fires (`t_streamclose`). Independently detect **DOM turn-end** (`t_domdone`): the stop/▮ button disappears AND the assistant message text is non-empty + stable for ~3 s (poll the DOM gently, ~1 s). 
   - **Verdict:** `PASS` iff exactly one such send-response stream is observed and `|t_streamclose − t_domdone|` is small (≤ ~3 s) and `t_streamclose` is not wildly before the visible answer completes — i.e. stream-close is a reliable turn-end signal needing ZERO completion poll. `FAIL` iff stream-close does not track turn-end (fires far from `t_domdone`, never fires, or the stream is ambiguous/multiplexed so close ≠ done). 
   - Close ONLY your own page; detach (do NOT quit the browser).
3. **Write** `team/evidence/handoffs/M18/probe-streamclose.md`: the single-token verdict (`PASS`/`FAIL`/`BLOCKED:<reason>`), the recorded **event table** (paths id-stripped only), the timeline (`t0_send`, stream path, `t_streamclose`, `t_domdone`, the delta), and a one-paragraph judgement. Echo `STATUS: <verdict>` + the path to stdout.

**Manager:** block on the probe worker; read the verdict file; **re-derive the PASS/FAIL yourself from the recorded timeline** (don't trust the worker's word). Then gate per `M18-common.md` (BLOCKED/FAIL → STOP + handoff; PASS → phase 2).
