# M3 Lens 4 — Concurrency: tab pool + adaptive rate (DESIGN ONLY)

**Read first, in full:** `team/contracts/M3-common-constraints.md`, then `docs/REWRITE-SPEC.md` (esp. §10 concurrency, §2 architecture — the persistent `Session` owns pool + budget), `team/evidence/handoffs/M2-ground-truth-probe.md`, `team/charter.md`, `.claude/skills/manager/references/agent-rigor.md` (esp. the **shared-resource-ceiling** rule — transcribe its logic into your design).

**Your lens:** design the **tab pool** and the **adaptive send-rate / account rate budget**, both single-owned by the persistent `Session`. Maximize concurrency without ever spamming the single operator account or tripping Cloudflare.

**Write your deliverable to:** `team/evidence/reports/M3-work/lens-4-concurrency.md` (begin with `STATUS:`).

## Produce these sections
1. **Managed tab pool.** Tabs are a **cache, not state** (stateless URL/id addressing makes re-opening an evicted conversation cheap). Design: lazy-open on demand; **idle-evict** (close tabs unused past an idle window); **LRU-reclaim** at a configurable generous ceiling. Concrete pool API: `acquire(conv_id) -> tab`, `release(tab)`, `evict_idle()`, `close_all()` (on detach). Map `conversation_id → tab`. Honor safety: **only tabs the tool opens** (never iterate `context.pages` / never touch operator tabs); **never quit the browser** (close own tabs + detach only). (REWRITE-SPEC §10.)
2. **Adaptive send-rate (not an artificial low cap).** Ramp up toward "as many as we can manage"; **back off on rate-limit/Cloudflare signals**; enforce a small **politeness floor** (min interval between sends). Define the limiter concretely (e.g. token-bucket + AIMD ramp/backoff): what are the **backoff trigger signals** (HTTP 429, Cloudflare "Just a moment"/challenge markers, error toasts, repeated send failures)? What is the politeness floor's default? Reads (scrape/backend-api capture) run **parallel**; **sends** are rate-governed. (REWRITE-SPEC §10.)
3. **Single-owner rate budget (shared-resource-ceiling).** The persistent `Session` is the **single owner** of the budget across all concurrent consumers. Apply the rigor rule: `sum(all concurrent consumers' send rates) + reserve ≤ account ceiling`, with a margin, re-derived whenever a consumer is added/removed; backoff is the safety **net**, not the primary control. The real account ceiling is **not yet empirically known** (M2 did not stress it) — design the limiter to **discover** it adaptively and flag that the numeric ceiling must be measured. Honor charter: **no hard message cap** (don't bake an arbitrary low number); human-paced + no spamming is the rule.
4. **Atomic-op vs persistent-Session budget coordination.** Atomic ops (single `ask`/`scrape`/`status`) each construct a short-lived `Session` (attach→act→detach) with no shared limiter to the persistent one — there is **no cross-process shared limiter unless built**. Discuss this gap honestly: atomic ops are one-shot/sparse (low collision risk); the persistent `Session` is where real concurrency lives. Recommend guidance (e.g. don't fan out many concurrent atomic-op processes; the ~3-way modest concurrency guidance from the charter). Do NOT over-engineer a cross-process coordinator unless you argue it's necessary (Occam).
5. **Concurrency profile.** Charter guidance: **modest, ~3-way** against the shared CDP browser (operator/another agent may be using it concurrently). State how the pool ceiling + send-rate defaults reflect this, and that they are configurable.

End with **"Cross-cluster interfaces & dependencies"** (pool/budget owned by `Session`/lens-1; reads from capture/lens-3 run parallel; sends from lens-1/lens-5 are governed) and **"Open questions / assumptions"** (esp. the unmeasured account ceiling + backoff-signal specifics that need M5/M7 confirmation). Occam: simplest limiter that protects the account and uses the full ceiling — no speculative distributed machinery.
