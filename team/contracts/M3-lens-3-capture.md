# M3 Lens 3 — Capture + auth-header mechanism + completion (LOAD-BEARING) (DESIGN ONLY)

**Read first, in full:** `team/contracts/M3-common-constraints.md`, then `docs/REWRITE-SPEC.md` (esp. §5 capture, §7 completion), `team/evidence/handoffs/M2-ground-truth-probe.md` (THE authoritative facts for this lens — read every line), `team/charter.md`, `.claude/skills/manager/references/agent-rigor.md`. Optional read-only reference: `issues/cdp-send-repro/controller.mjs`.

**Your lens is the riskiest, highest-criticality piece of the whole design.** M2 confirmed the backend-api capture works but **only with the web-app auth/OAI headers** (cookies-only → 404). The auth-header mechanism + the fail-closed fallback are why this lens is load-bearing. Be exhaustive and concrete; enumerate options with trade-offs; recommend one.

**Write your deliverable to:** `team/evidence/reports/M3-work/lens-3-capture.md` (begin with `STATUS:`).

## Produce these sections
1. **Auth/OAI-header acquisition — THE load-bearing mechanism.** Design exactly how the tool obtains the header set the web app uses for `GET /backend-api/conversation/<id>` (`authorization` bearer, `oai-client-build-number`, `oai-client-version`, `oai-device-id`, `oai-language`, `oai-session-id`, `x-openai-target-path`, `x-openai-target-route`). **Enumerate the candidate options with trade-offs and pick one:**
   - (a) **Intercept the page's OWN request** (RECOMMENDED, per spec + M2): on the tool's own attached tab, register a CDP/Playwright network listener (`page.on("request")` / route interception / `request.all_headers()`), navigate the tab to the conversation URL (allowlist-checked) so the app issues its own `/backend-api/conversation/<id>` request, harvest the headers from THAT request, then issue the capture fetch with them.
   - (b) Read the bearer from the page JS/session store — note fragility + over-reading risk.
   Specify the concrete Playwright/CDP calls, how you wait for the app's request, and how you handle the app not issuing one. **Never persist or log any header value** (esp. `authorization`) — describe the in-memory-only handling and that it must never reach `raw-mapping.json`, `transcript.jsonl`, logs, or error text.
2. **Capture pipeline (happy path).** End-to-end: allowlist-check URL → acquire tab (from pool) → harvest headers (§1) → issue `GET https://chatgpt.com/backend-api/conversation/<id>` as an **in-page `fetch`** (`page.evaluate`) in the operator's session with the headers → receive the JSON → persist `raw-mapping.json` → parse → linearize → hand records to the store. Note it returns **~17.1 MB, ~5k nodes, no pagination** (M2-measured).
3. **Memory/streaming strategy (rigor: O(batch), measure don't guess).** 17 MB is measured. Reason explicitly about peak memory of parsing a 17 MB JSON into Python (state your estimate and that it must be **measured in M5**, not assumed). Recommend a stream-then-persist approach: stream the response body to `raw-mapping.json` on disk first, then parse (incrementally if needed, e.g. `ijson`, or whole-file if the measured footprint is acceptable). Avoid any "enumerate everything in memory then process" anti-pattern. Reads run in parallel (concurrency is lens 4's; just state the dependency).
4. **`mapping` → canonical markdown extraction.** Per-node rule keyed by `(author_role, content_type)`: `assistant:text` → join `message.content.parts` (list of strings); `assistant:code` → `message.content.text`; `assistant:thoughts` → `message.content.thoughts` (HIDDEN — retained in raw only, not in the linear transcript); `tool:*`, `assistant:reasoning_recap`, `assistant:model_editable_context`, `system:*` → hidden. Define the **visible-vs-hidden** classification the linearizer (lens 2) consumes. DR/Pro: visible report = the `assistant:text` in the `turn_exchange_id` group; citations from `metadata.content_references`/`citations`/`search_result_groups`/`search_queries`.
5. **Fail-closed fallback chain (gotcha #1).** Define the **exact trigger conditions** (auth headers unobtainable; HTTP 404/403; backend JSON shape unrecognised/unparseable) and the chain: backend-api canonical markdown (primary) → per-turn **copy button** → clipboard raw markdown (grant-only, attended — NOT an unattended fallback per M2) → KaTeX `<annotation encoding="application/x-tex">` reconstruction → DOM `textContent` (known-lossy, last resort). **Fail-closed** = on fallback, flag/raise (`status`/error), never silently emit corrupt/ambiguous math. Fidelity bar: `\widehat`, `\ne`, `\frac{}{}` round-trip vs web-UI **copy** output — verified in M5/M6, never assumed-by-construction.
6. **Completion detection (gotcha #3, no hidden ceiling).** **Primary:** poll the authenticated conversation endpoint until the new turn is complete — using top-level `async_status`, metadata `is_complete`/`is_finalizing`/`pro_progress`, node `status`; consider `GET /backend-api/conversation/<id>/stream_status` (hypothesis — flag for M5 verification). **Fallback (DOM consensus, ported from v1/controller.mjs as reference):** streaming-marker seen → stop-button absent ≥ stable window → text stable ≥ window → (saw-streaming OR non-empty body), **gated on the new-turn baseline** (different `message_id` than baseline). **`timeout` is a no-activity window that resets on progress — NOT a hard cap**; optional `max_total_wait` defaults unbounded; long Pro/DR runs (minutes) must never be silently killed. On timeout/error, **salvage** partial visible text (hand to store with `partial=true`).

End with **"Cross-cluster interfaces & dependencies"** (records/fields you hand to store/lens-2; how send/ask in lens-1 invokes completion; tab/rate from lens-4) and **"Open questions / assumptions"** (esp. anything that can only be confirmed against the live site in M5). Occam: prefer the simplest mechanism that is correct and safe.
