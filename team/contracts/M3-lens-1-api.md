**Write your deliverable to:** `team/evidence/reports/M3-work/lens-1-api.md` (begin with the `STATUS:` line per common §6).

## Produce these sections
1. **`Session` object — full API.** Constructor params (`cdp_endpoint`, `data_dir`, `channel` in {`mock`,`cdp`}, config knobs) and every method with **typed signatures + one-line semantics**: `create(project=None)`, `ask(conv_or_url, prompt, *, model=None, tools=(), attach=(), timeout=..., max_total_wait=None, out=None)`, `scrape(conv_or_url, *, with_attachments=False)`, `history(conv_or_url)`, `fetch(conv_or_url, attachment_ref)`, `status(conv_or_url=None)`, `loop(...)`, plus lifecycle (attach/detach, context-manager `__enter__/__exit__`). Mark which touch the browser. (REWRITE-SPEC §3, §4.)
2. **Result/return object model.** Define what `ask`/`scrape` return — e.g. a `TurnResult`/`Transcript` dataclass exposing `content_markdown`, `message_id`, `parent_id`, `role`, `model`, `active_tools`, `kind`, `created_at`, `attachments`, `citations`, `status`, `partial`. This must align with the JSONL record (lens 2) and the capture output (lens 3) — state the alignment explicitly.
3. **Atomic-op vs persistent-Session composition.** How atomic CLI verbs build a short-lived `Session` (attach→act→detach) while `loop`/concurrent use hold one open; why the persistent `Session` is the single owner of the tab pool + rate budget (defer the pool/rate internals to lens 4 but define the ownership boundary).
4. **Complete module list** — for each: `session.py`, `capture.py`, `send.py`, `completion.py`, `menus.py`, `store.py`, `identity.py`, `channels/{base,mock,cdp}.py`, `selectors/` (`mock.json`,`real.json`), `errors.py`, `allowlist.py`, `cli.py` — give a one-paragraph **responsibility** + the **key public signatures** it exposes. (REWRITE-SPEC §2 module layout.)
5. **Channel abstraction (the browser seam).** Define the interface that BOTH `mock` and `cdp` implement (the methods `Session` calls instead of touching Playwright directly — e.g. `attach()`, `open_tab(url)`, `evaluate(js)`, `fill(sel,text)`, `click(sel)`, `query_turns()`, `fetch_in_page(url, headers)`, `read_clipboard()`, `detach()`). This is what lets the full offline mock suite exist. Specify what the `mock` channel must simulate deterministically.
6. **`send.py` mechanics (verified-send gotcha #2).** The send flow: baseline latest user-turn `message_id`/count → fill `#prompt-textarea` (insertText fallback) → submit (send button, Enter fallback) → poll for a NEW user turn carrying the prompt → else raise `PromptNotSubmittedError` → wait/retry the transiently-unmounting composer → reload-when-idle to clear SPA staleness. Model/tool selection verified before send. (REWRITE-SPEC §6; controller.mjs is the real-proven reference.)
7. **`menus.py` (label-driven Radix).** The one general mechanism for model picker + tools/`+` menu: open trigger → enumerate `[data-radix-popper-content-wrapper]` → select by display label → verify reflected → fail-closed. Tiers `menuitemradio`, families `menuitem` submenu. Model picker has NO stable test-id (label-driven). (REWRITE-SPEC §11; M2 selectors.)

End with **"Cross-cluster interfaces & dependencies"** and **"Open questions / assumptions"** (common §6). Keep it the simplest correct skeleton (Occam) — no daemon, no speculative abstraction.
