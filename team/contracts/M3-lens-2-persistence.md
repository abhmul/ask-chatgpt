# M3 Lens 2 — Persistence + schema (DESIGN ONLY)

**Read first, in full:** `team/contracts/M3-common-constraints.md`, then `docs/REWRITE-SPEC.md` (esp. §8 persistence, §9 identity), `team/evidence/handoffs/M2-ground-truth-probe.md` (the JSON shapes you must represent), `team/charter.md`, `.claude/skills/manager/references/agent-rigor.md`.

**Your lens:** design the **on-disk persistence layer + the transcript schema** — the lossless, resumable spine. The pressing deliverable (scraping the ~17 MB / ~5k-node math conversation) lands here, so the schema must faithfully represent **every M2 JSON shape**.

**Write your deliverable to:** `team/evidence/reports/M3-work/lens-2-persistence.md` (begin with `STATUS:`).

## Produce these sections
1. **Data-dir layout.** Concrete tree under `--data-dir`/`ASK_CHATGPT_DATA_DIR` (default `~/.local/state/ask-chatgpt/`): `index.json`; `conversations/<conversation-id>/{transcript.jsonl, raw-mapping.json, attachments/, .gitignore}`. State what each file holds + that `attachments/` is gitignored + lazily filled. (REWRITE-SPEC §8.)
2. **`transcript.jsonl` per-turn record — full schema.** Every field with **type + semantics + source**: `conversation_id`, `message_id` (canonical; idempotency key AND send-verification baseline), `parent_id`, `turn_index`, `role`, `content_markdown` (canonical, untruncated), `model` `{slug, display}` (independent of tools), `active_tools[]` (orthogonal to model — DR is a tool), `kind` (`normal|deep_research|image|...`), `created_at` (from backend-api timestamp — **never an agent self-report**), `attachments[]`, `citations[]`, `status` (`complete|partial|error`), `partial`. Append-only; reads do last-writer-wins per `message_id`.
3. **Tree → current-branch linearization.** The algorithm: backend-api `mapping` is a message **tree**, `current_node` is the leaf; walk `parent` links from `current_node` to root, reverse, emit one record per **visible** message (visible-vs-hidden classification is lens 3's; consume its rule). Retain the **full** tree losslessly in `raw-mapping.json`. Branch-aware history is deferred (raw tree retained meanwhile). (REWRITE-SPEC §8 conversation tree; M2 recommendation #4.)
4. **Deep-Research / Pro turn representation.** How a DR/Pro turn (large `turn_exchange_id` group: one user msg + many hidden `assistant:thoughts`/`code`/`reasoning_recap` + tool nodes + one visible final `assistant:text` report in `content.parts[0]`) maps to records: the visible report becomes the transcript record; hidden internals stay in `raw-mapping.json`; `kind="deep_research"`; `active_tools` includes the DR tool; citations captured. Decide whether/how to record the `turn_exchange_id` for grouping. (M2 "Deep Research representation".)
5. **Attachment representation — cover ALL M2 shapes.** A unified attachment record `{filename, mime, bytes, sha256, source_ref, local_path:null}` and a mapping table from each observed M2 shape into `source_ref`: `message.metadata.attachments[]` (user uploads); `message.metadata.content_references[type=file]`; `message.content.assets[].asset_pointer` (generated/image); `message.metadata.aggregate_result` (code-exec outputs). No literal `/files/` URLs were seen — references are ids/pointers; bytes are downloaded **lazily** by `fetch`. Keep **citations** (DR web-source `{title,url}` from `content_references`/`citations`) **separate** from attachments — citations are never downloaded. (M2 "Attachments reference shape"; REWRITE-SPEC §8 attachments-vs-citations.)
6. **`index.json` schema.** alias/session-id → `conversation_id`, `model`, `project_id`, `title`, `last_updated`. Stateless URL/id addressing means index is a convenience, not required to address a chat. (REWRITE-SPEC §9.)
7. **Write discipline (lose-nothing, gotcha #3).** Record lifecycle: **eager-write** a stub (prompt + conversation ref) at/just-before send → update with full response on completion → on error/timeout **salvage** partial visible text with `status` + `partial=true`. **Atomic write mechanics** (temp-file + rename, or append + fsync) so a crash mid-write never corrupts the file. Idempotent re-scrape (last-writer-wins per `message_id`).

End with **"Cross-cluster interfaces & dependencies"** (what fields you require capture/lens-3 to deliver; what the API/result object lens-1 reads) and **"Open questions / assumptions"**. Occam: the schema should be the minimum that loses nothing — flag any field you suspect is over-engineered.
