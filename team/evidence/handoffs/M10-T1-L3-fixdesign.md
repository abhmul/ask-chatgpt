## Answers to required lens questions

1. Light-acquire API: prefer `TabPool.acquire(ref, *, render: bool = True)` over a `channel.open_tab(render=...)` parameter. The channel already opens a supplied URL; adding a render flag there is unnecessary complexity. Pool keying must be by `(mode, url)`, not URL alone, because a light root tab (`("light", "https://chatgpt.com/")`) would otherwise collide with draft send/root tabs and with any root-keyed entry whose actual SPA URL later client-navigated. One shared light root tab can serve reads for any persisted conversation in a single `Session`, because backend fetches are same-origin and conversation id is carried in the fetch path, not in the page URL; if ambient headers cannot be re-harvested from a reused light tab, reload the light root or use an in-memory header cache with stale-header invalidation, but never navigate to `/c/<id>` for read harvest.
2. Exact files/lines to edit: `session.py:44-48` (`_ManagedTab`) add `key`; `session.py:82-97` add the `render` parameter, light URL selection, and `(mode,url)` cache comparison; `session.py:521-523` make `scrape` acquire light and pass `header_mode="ambient_backend"`; `capture.py:55-68` optionally extend `HeaderBundle` with observed path / target-path helper; `capture.py:140-162` add `mode` to header acquisition and ambient `/backend-api/` matcher; `capture.py:305-307` pass the mode and retarget copied headers for `/backend-api/conversation/<id>` before `stream_backend_conversation`; `completion.py:45-51` should keep exact/default harvest, but if the `for_single_fetch` signature changes, update it mechanically without changing behavior; `channels/base.py` and `channels/cdp.py` need no signature change. `channels/mock.py` only needs test support if strict target-path behavior is modeled.

## Risks / unknowns

- The fix makes renderer/page work O(1) in conversation size; the backend raw JSON fetch and parse still scale with conversation length. Do not overclaim total read cost as O(1) bytes or CPU.
- Live `x-openai-target-route` retarget syntax is unknown from committed evidence. If keeping route verbatim works, that is the smallest safe change; if real root harvest returns a route rejected by conversation fetch, implementation needs a live-derived retarget rule or another safe header source.
- A reused light tab may not naturally emit fresh backend requests after its initial load. The implementation must either cache transient headers in memory with fail-closed invalidation or reload only the light root on cache miss; it must not silently fall back to `/c/<id>`.
- CLI tab leak and rate limiting remain separate issues. M10 should not block on them, but M10 verification must avoid adding bursty requests.
