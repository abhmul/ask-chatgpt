## Answers to required questions

1. Harvest predicate: `acquire_backend_headers` computes `target_path = urlsplit(backend_conversation_url(conversation_id)).path`, where the backend URL helper returns `https://chatgpt.com/backend-api/conversation/<id>` (`src/ask_chatgpt/capture.py:140-142`; `src/ask_chatgpt/identity.py:127-130`). `matches()` accepts only `request.method.upper() == "GET"` and `path == target_path` (`src/ask_chatgpt/capture.py:144-146`). `BrowserChannel.wait_for_request` is a protocol method (`src/ask_chatgpt/channels/base.py:81-87`); CDP implements it by scanning the owned tab’s request buffer, applying the predicate first to a cheap URL/method snapshot, then collecting required headers from Playwright `all_headers()` and CDP ExtraInfo until all `REQUIRED_CAPTURE_HEADERS` are present or the timeout expires (`src/ask_chatgpt/channels/cdp.py:754-793`, `src/ask_chatgpt/channels/cdp.py:938-955`). This is passive: it does not trigger a request. The SPA issues the exact conversation GET as route data for `/c/<id>`; M7b proves fresh/root/client-held state does not issue it until the tab is reloaded on `/c/<id>` (`team/evidence/handoffs/M7b-gaps.md:16`; `src/ask_chatgpt/channels/mock.py:305-310`).

2. Header conversation-agnosticism classification: `authorization` = session/auth-global; `oai-client-build-number` = app-build global; `oai-client-version` = app-version global; `oai-device-id` = device global; `oai-language` = user/app preference global; `oai-session-id` = browser/app-session global. These six are not named for a URL path and M2’s evidence ties their necessity to authenticated web-app requests generally (`src/ask_chatgpt/capture.py:40-45`; `team/evidence/handoffs/M2-ground-truth-probe.md:13-16`). `x-openai-target-path` and `x-openai-target-route` = request/path-specific for design purposes: their names encode target routing, the current code treats them as opaque harvested values, and no inspected code/evidence proves cross-path reuse from a non-conversation request is safe (`src/ask_chatgpt/capture.py:46-47`, `src/ask_chatgpt/capture.py:62-69`, `src/ask_chatgpt/capture.py:174-180`). M2 only proves conversation-path values replayed to the same conversation endpoint work (`team/evidence/handoffs/M2-ground-truth-probe.md:14-16`).

3. `for_single_fetch`: it does not recompute either target header. In the usual capture path, `capture_conversation` consumes the bundle to a plain mapping (`src/ask_chatgpt/capture.py:305-307`), and `stream_backend_conversation` copies or consumes headers and sends them unchanged except for adding `accept: application/json` (`src/ask_chatgpt/capture.py:174-180`). Harvesting from `/backend-api/me`, `/backend-api/models`, or any non-conversation request would therefore send that request’s target headers verbatim to `/backend-api/conversation/<id>` unless implementation changes.

4. Light-page request inventory: code-certain inventory is effectively none for root auto-requests. Source hardcodes only conversation, conversation `stream_status`, and files/download backend paths (`src/ask_chatgpt/identity.py:130`; `src/ask_chatgpt/completion.py:45-51`; `src/ask_chatgpt/capture.py:176-180`; `src/ask_chatgpt/capture.py:422-429`), and root draft refs only identify `https://chatgpt.com/` as a light page (`src/ask_chatgpt/session.py:337-345`). M2 confirms a fresh page exists and accept-only conversation fetch fails (`team/evidence/handoffs/M2-ground-truth-probe.md:13`), but it does not list root SPA requests. M7b selector probes prove the root composer/menu UI can mount, not which backend requests it emits (`team/evidence/reports/M7b-T1-selectors.md:8-18`). Candidate live root requests such as `/backend-api/me`, `/backend-api/models`, `/backend-api/conversations`, and `/backend-api/accounts/...` are plausible but not established by the inspected evidence. Whether at least one reliably carries all eight required headers is only confirmable by an attended real-site probe.

5. Minimal harvest change: option (a) is the smallest viable design if the real probe is positive: open `https://chatgpt.com/`, harvest from any same-origin `GET /backend-api/*` that carries all eight required names, then recompute `x-openai-target-path` and `x-openai-target-route` for `/backend-api/conversation/<id>` before the conversation fetch. This also needs a wait-layer tweak: current `wait_for_request` can return the first header-deficient matching request after waiting (`src/ask_chatgpt/channels/cdp.py:774-793`), so an “any backend request carrying all 8” harvester must skip deficient matches or add a header-aware predicate/continuation. Option (b), actively triggering a harvestable request from root, is not minimal unless it uses ordinary root SPA behavior; an injected bare `fetch()` will not add the auth/OAI headers, as M2’s accept-only 404 shows (`team/evidence/handoffs/M2-ground-truth-probe.md:13-15`). I recommend (a) after the real probe; do not ship a root-light scrape on the current exact-conversation harvest.

## Risks / unknowns

- The exact `x-openai-target-route` recomputation rule is unknown offline; do not log header values. The real probe should compare values in memory against public candidate paths/routes and report booleans/categories only.
- Root SPA request inventory and all-8-header reliability may vary by account state, region, feature flags, sidebar state, and app version. M7b/M2 do not settle this.
- If root’s first `/backend-api/*` request lacks one of the eight headers, the current wait implementation would fail early for an any-backend predicate unless changed to continue scanning.
- No `uv run pytest` was run; this was read-only code/evidence investigation with no source/test edits.
