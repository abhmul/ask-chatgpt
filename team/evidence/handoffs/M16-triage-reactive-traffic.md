## 3. Major issue — what's needed (change set, priority order)
The issue's optimistic "the primitives exist, it's mostly surgical" framing is **mostly right on the verdict but wrong on three specifics** (see §3.6). Change set, highest human-like impact first:

### C-1 — Persistent tab + kill the per-send reload  →  {surgical + **moderate refactor**}
- **Surgical part:** drive multi-turn via the existing persistent-tab path. `Session.loop` already acquires a tab **once** (`session.py:583`) and reuses it across iterations (`session.py:586-604`), vs `ask` re-acquiring per call (`session.py:370`) and the CLI detaching/closing per process (`cli.py:206` → `session.detach` → `tab_pool.close_all` `session.py:326-328`/`112-119`). So "stop spawning a process per turn" is a driver/usage change, not a v2 change.
- **Moderate refactor (the part the issue misses):** "drive via `loop`" does **not** by itself yield "one page-load per session." `_run_send_turn` (shared by `ask` and `loop`) calls `wait_for_idle_and_reload_if_needed` (`session.py:404` → `send.py:48`), which **reloads the page on every send** when idle (`send.py:54-56`), plus a draft post-completion reload (`session.py:455`). To hit the acceptance criterion you must split idle-wait from reload (reload only when actually mid-generation / on a real staleness signal), in `send.py:48-63` and `session.py:455`.
- **Header-harvest dependency (verifier caveat — do not miss):** the per-send/draft reload is **not purely** a page-load amplifier — it also generates the observable `/backend-api/*` request that `acquire_backend_headers` harvests the 8 `REQUIRED_CAPTURE_HEADERS` from (`capture.py:154-190`; the conversation matcher accepts only matching `GET`s `capture.py:943-947`; `tests/test_session_draft_loop.py:270-292` already pins a pre-capture reload when the mock requires it). So the reload-split must **preserve a header-harvest trigger** — harvest opportunistically from the new Network/stream observer (C-2), or keep a draft-only reload — and **must not** delete `session.py:455` blindly. Gate this with the offline harvest test in §5.
- **Heavy-render caveat:** sends use the render tab (`/c/<id>`, `session.py:86,370,583`); a persistent render tab on a huge conversation re-introduces the renderer OOM (memory `read-via-light-page-not-render`, `driver-reap-closes-operator-tab`; issue line 99). Pair persistent-tab with **rotate-before-heavy** (the rig already rotates). Reads stay on the M10 light path (`scrape` uses `acquire(render=False)` `session.py:525`).

## 4. Reactive-completion feasibility
**Verdict: NEEDS-NEW-NETWORK-OBSERVER** (new code on existing infra; NOT a re-architecture), **gated on a 1-turn attended probe.**

- **Today the send observes no network stream:** `submit_composer` (`send.py:178`) just clicks; `verify_prompt_submitted` (`send.py:212`) only polls the DOM for a new user turn; completion is a *separate* poll loop. The SPA's own send-response stream is never watched.
- **Seam (a) binding-stream — CANNOT observe the SPA stream.** `JS_STREAM_FETCH` (`cdp.py:50-84`) runs `const response = await fetch(url, init)` — a fetch **the tool itself injects** — and emits `{kind:"done"}` (`cdp.py:78,509`) when *that tool fetch's* body ends. Used only by `fetch_in_page(stream_to=...)` (`cdp.py:813-849`) for backend capture/attachment download. It is not a hook into the SPA's native fetch/EventSource.
- **Seam (b) Network-observation — the right infra, partially pre-wired, but no finish-waiter.** `_install_page_observers` already registers `page.on("requestfinished", …)` (`cdp.py:886`) and `_install_cdp_observers` opens a CDP session on `Network.requestWillBeSent` (`cdp.py:889-902`). But `wait_for_request` (`cdp.py:754`) consumes only the request-**sent** buffer (header harvest), and **`_record_page_request_finished` is a literal no-op** (`cdp.py:917-918`: `del tab_id, request`). So the finish event is already delivered to the tool and **thrown away** — the new observer is a small, well-located addition: record `requestId → path/method/status/type` and wait for `Network.loadingFinished`/`loadingFailed`/`requestfinished` (and optionally `eventSourceMessageReceived`) on the identified conversation stream request.
- **Minimal attended probe (designed, NOT run):** attach to the operator's CDP browser; `Network.enable`; record only `{method, path, status, eventName, requestId, monotonic-ts}` (never bodies/headers-values/text). Mark `t0` just before the send click; identify the `/backend-api/*` stream request starting after `t0`; confirm its `loadingFinished`/`requestfinished` fires **when and only when** the assistant turn ends (with a 3-5 s DOM-stability guard after). Pass = exactly one native stream whose close coincides with turn-end, needing zero `/backend-api/conversation/<id>` poll to decide completion.
- **Robustness + fallback:** a pure stream-close signal is **more** robust than today's DOM heuristic for SHORT multi-turn responses (ties completion to the server response actually ending), but **less** robust as a *sole* signal under reconnects/multiplexing/early-close. **Keep a DOM-stability fallback** + a low-rate backend rescue (not the default). NOTE: the actual code heuristic is `stable_window_s = progress_poll_interval_s = 2.0 s` (`completion.py:147,135`) with a `stop_absent`+new-id+non-empty gate (`completion.py:401-405`) — there is **no** separate "≥3 s + global-marker" path in `completion.py` (the memory `real-completion-clips-short-multiturn` describes an earlier design; re-derive the fallback from current code).

## 6. Rate-governor + 429 design
- **Shared-resource ceiling (agent-rigor):** all account requests (sends + page-loads + fetches + uploads + the operator's own browsing + any other agent) share **one** per-account ceiling; independent per-process limiters can each be "under limit" while their sum exceeds it. Invariant: `Σ(consumer rates) + reserve ≤ C_account·(1 − margin)`. **`C_account`, the reserves, the per-action token weights, and the margin are OPERATOR-OWNED** — they must be **confirmed with the operator (or measured in the attended baseline and then confirmed), never guessed or inherited** (memory `verify-inherited-resource-claims`). The site rule is human-paced + no programmatic spamming, **not** a message cap (memory `real-site-no-message-cap`; `AdaptiveSendBudget` already encodes "no hard message cap" `session.py:140,210-214`).
- **Cross-process home (the hard part — M12 confirmed):** `AdaptiveSendBudget` is per-`Session`, **in-memory** (`session.py:139,302`), **send-only** (governs only the `submission()` block `session.py:413`; not page-loads/fetches), `record_soft_signal` is **unwired** (no caller besides the def `session.py:194`), and a fresh `ask`/`scrape` process gets no first-send spacing (`session.py:224-225`). **Recommendation:** a **file-lock token bucket** in a shared governor dir (default under `Store.resolve_data_dir()` `store.py:48`; reuse the existing `fcntl.flock` pattern `store.py:390`), coordinating all tool processes; a long-lived `loop` is a *consumer* of the bucket (good for page-load reduction) but cannot be the sole owner (standalone `scrape`/`ask`/other agents bypass it). State holds only non-secret ops data (tokens, refill epoch, `blocked_until`, last-rate-limited, redacted action/path kind). `acquire(cost, action, path_kind)` refills by the operator-confirmed allocation, debits, and either proceeds / sleeps / raises if `blocked_until` is set.
- **Chokepoints to wrap (no single `fetch_in_page` covers them):** new tab/page-load (`TabPool.acquire`→`open_tab` `session.py:98`, `cdp.py:635-649`); reloads (`send.py:55`, `session.py:455`, `cdp.py:672-674`); SPA send click (`send.py:178`, `session.py:418`); explicit backend fetches (completion `completion.py:47`, capture `capture.py:205`, attachments `capture.py:446,461`); uploads (`send.py:123`, `cdp.py:867-869`). Header-harvest *observation* (`capture.py:154`) is not charged — the page-load/fetch that produced the observable request is charged at its own chokepoint.
- **429 / "Too many requests" fail-soft:** add `RateLimitedError` to `errors.py` (near the timeout family `errors.py:212-223`) with **exit code 52** (verified free — `_ERROR_EXIT_BY_CODE` `cli.py:29-49` jumps 42→50; LENS-4's draft used `43`, **standardize on 52**), `retryable=True`, `retry_action="back_off"`; add `RATE_LIMITED:52` to the CLI table and ensure it takes the plain error path (no clipboard salvage). Detect 429 **before** `BackendCaptureShapeError`/fallback: in `poll_backend_completion` (`completion.py:54-58`, and do **not** add it to the swallowed set `completion.py:160-164`) and at the capture fetch boundary (`capture.py:205-216`/`936-938`, **not** caught by the fallback `capture.py:358-360`). Parse only the `Retry-After` **seconds** (the value boundary already strips secrets — `cdp.py:362-370` preserves `retry-after`); never log raw header values. Wire `record_soft_signal('rate_limited')` + the shared `blocked_until` from a new `except RateLimitedError` in `_run_send_turn` (`session.py:469-480`). Distinguish a **volume 429** (retryable backoff; default ~30 min silence on a header-less modal — memory `chatgpt-rate-limit-guidance`) from **quota/plan exhaustion** (classified separately; pacing won't create quota).
- **Human pacing + jitter:** keep `AdaptiveSendBudget` as the local pacer but add injectable-RNG **jitter** to `_required_spacing_s`/`_sleep_until_spacing_allows_submit` (`session.py:220-229`); prefer **one-sided** extra delay (never undercuts the global cap); the cross-process bucket supplies the spacing that the per-process first-send no-op cannot.
