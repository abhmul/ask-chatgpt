## Shared-resource ceiling and allocation owner

All ChatGPT web-account requests share one per-account ceiling: sends, SPA page-load bursts, explicit backend fetches, attachment upload/download, and operator or other-agent browsing. Independent per-process limiters can each be under limit while their sum exceeds the account ceiling.

Use this invariant: `lambda_tool_driver + lambda_tool_ad_hoc + lambda_operator_browser_reserve + lambda_other_agents_reserve + lambda_emergency_reserve <= C_account * (1 - margin_fraction)`. Equivalently, `sum(consumer rates) + reserve <= confirmed ceiling`, with reserve including the safety margin. `C_account`, reserves, page-load weights, send weights, burst size, and margin are OPERATOR-OWNED policy values: they must be confirmed with the operator or measured in an attended request-count baseline and then confirmed. They must not be guessed or inherited from old prose. The site rule here is human-paced and no programmatic spamming, not a fixed message budget.

The single owner of the allocation should live in an account-level policy file, not in `Session`: e.g. `<governor-dir>/policy.json`, defaulting to `Store.resolve_data_dir()/rate-governor/`; `Store.resolve_data_dir()` is defined at `src/ask_chatgpt/store.py:48-57`. Code may update `<governor-dir>/state.json`, but only the operator or one supervisor edits policy allocation. If multiple data dirs are used for the same logged-in account, set one shared governor dir such as `ASK_CHATGPT_GOVERNOR_DIR`; otherwise separate data dirs become separate uncoordinated limiters.

## Recommended cross-process mechanism

Recommend a file-lock token bucket under the shared governor dir, plus long-lived `loop` for page-load reduction. The file-lock bucket is the minimal mechanism that coordinates fresh `ask`, fresh `scrape`, `loop`, and future library callers. A long-lived `loop` owner is still valuable for Layer B/C-1, but it is not sufficient as the global governor because standalone commands, another process, or another agent can bypass it.

| Option | Fit | Trade-offs |
|---|---|---|
| File-lock token bucket under shared data/governor dir | Coordinates all tool processes that share the dir; survives process restart; can block or fail before every request-emitting action. Existing store code already uses `fcntl.flock` for cross-process transcript locking at `src/ask_chatgpt/store.py:383-393`. | Cannot directly police manual operator browsing; that must be reserved in policy. Page-load bursts need measured and confirmed token weights because one `open_tab` triggers multiple SPA requests. |
| Long-lived `loop` owns budget | `loop` acquires one tab at `src/ask_chatgpt/session.py:583` and iterates at `src/ask_chatgpt/session.py:586-604`, so it avoids the fresh-process first-send reset and per-turn page load. | Does not coordinate `scrape`, ad-hoc `ask`, other agents, or operator browsing; state dies with the process. Use it as a consumer of the file-lock bucket, not the sole owner. |

Token-bucket state should contain only non-secret operational data: tokens, last refill epoch, blocked-until epoch, last rate-limited time, and redacted action or path categories. `acquire(cost, action, path_kind)` takes the lock, applies any `blocked_until`, refills by the operator-confirmed tool allocation, debits tokens, and either proceeds, sleeps until affordable, or raises `RateLimitedError` if blocked by a previous 429. On 429, `note_rate_limit(retry_after_s)` sets `blocked_until_epoch_s = max(existing, now + retry_after_s_or_default)` so all processes observe the backoff.
