# M16 Lens 2 — surgical-vs-rewrite architecture verdict

Status: DONE

## Overall verdict

**SURGICAL**, not a rewrite of v2, but not the issue's optimistic "just reuse existing primitives" version. The existing session/send/capture split can absorb the fix set through localized patches plus two moderate refactors: (1) remove/replace the per-turn reload behavior while using the existing persistent `loop`/`TabPool` model, and (2) add a new CDP native-network stream-close observer because the existing `StreamState` streams only tool-owned `fetch()` calls. I would not implement optional capture-from-stream in the first fix; making the native stream replace backend capture is the only item that looks like major rework.

Three strongest reasons:

1. The persistent-tab model already exists: `TabPool.acquire()` reuses unleased tabs for the same key at `src/ask_chatgpt/session.py:84-100`, `Session.loop()` acquires once at `src/ask_chatgpt/session.py:583` and releases at `src/ask_chatgpt/session.py:606`, while one-shot CLI `ask` creates a session at `src/ask_chatgpt/cli.py:188-206` and detaches/close-all via `src/ask_chatgpt/session.py:326-328` and `src/ask_chatgpt/session.py:112-119`. That is a lifecycle/driver usage problem, not a new session architecture.
2. Completion waiting is centralized enough: `_run_send_turn()` calls `wait_for_completion()` once at `src/ask_chatgpt/session.py:444-452`, and the poll loop/partial attachment live in one function at `src/ask_chatgpt/completion.py:127-206` with `_attach_partials()` at `src/ask_chatgpt/completion.py:461-466`. Adding a stream-close path has ripples into channel/mock plumbing and partial salvage, but not into store/capture identity semantics.
3. The request governor can be added around existing BrowserChannel/Session seams (`open_tab`, `reload`, `fetch_in_page`, submit click) rather than rewriting send/capture. The hard part is cross-process state, because the current `AdaptiveSendBudget` is per-`Session` at `src/ask_chatgpt/session.py:302`, send-only at `src/ask_chatgpt/session.py:413`, and has no caller for `record_soft_signal()` beyond its definition at `src/ask_chatgpt/session.py:194`.

## Where the issue's "the primitives exist" claim is wrong

1. **`StreamState` is not the native SPA stream observer.** `JS_STREAM_FETCH` defines its own in-page `fetch(url, init)` at `src/ask_chatgpt/channels/cdp.py:50-84`, reads that response body at `src/ask_chatgpt/channels/cdp.py:66-78`, and emits `{kind: "done"}` only for that tool-owned fetch. `fetch_in_page(stream_to=...)` creates the binding/state at `src/ask_chatgpt/channels/cdp.py:813-823`, evaluates `JS_STREAM_FETCH` at `src/ask_chatgpt/channels/cdp.py:837`, and checks `stream_state.done` at `src/ask_chatgpt/channels/cdp.py:847-849`. The native composer submit at `src/ask_chatgpt/send.py:178-204` is just a click; it does not route the SPA's response stream through `StreamState`. The existing CDP observer subscribes only `Network.requestWillBeSent` and `Network.requestWillBeSentExtraInfo` at `src/ask_chatgpt/channels/cdp.py:900-902`, so C-2 needs a new response/finish observer rather than reusing binding-stream as-is.
2. **Persistent tab is not sufficient while reload-on-idle remains.** Even in `loop`, `_run_send_turn()` calls `wait_for_idle_and_reload_if_needed()` every iteration at `src/ask_chatgpt/session.py:404`, and that helper reloads immediately when the stop button is absent at `src/ask_chatgpt/send.py:54-56`. Therefore the current code still creates a per-turn page reload unless this helper is changed.
3. **The governor has no cross-process or all-request home.** `AdaptiveSendBudget` is per `Session` and instantiated in memory at `src/ask_chatgpt/session.py:302`; one-shot CLI processes create/detach sessions in `src/ask_chatgpt/cli.py:188-206`; the only enforcement is the send submission context at `src/ask_chatgpt/session.py:413`; repo search finds no caller of `record_soft_signal()` beyond `src/ask_chatgpt/session.py:194`. A real governor must coordinate outside a single process and account for page-load/reload/click-triggered SPA requests.

## Flip conditions

1. If the attended probe shows the native send stream has no reliable CDP close signal (`Network.loadingFinished`, equivalent EventSource/Fetch lifecycle, or another stable request-completion marker), C-2 escalates from moderate refactor to major completion-model work.
2. If acceptance requires eliminating the post-completion backend capture GET and deriving a canonical transcript with citations/attachments solely from the native stream, C-3 becomes major rework and should be split from the rate-limit fix.
