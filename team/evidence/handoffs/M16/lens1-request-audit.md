## Completion poll path: verified correction

The issue’s Finding #2 is wrong on the polled path. `poll_backend_completion` has `prefer_lightweight: bool = False` (`src/ask_chatgpt/completion.py:42`). The path expression is `/backend-api/conversation/<id>/stream_status` only if `prefer_lightweight` is true; otherwise it is `/backend-api/conversation/<id>` (`completion.py:46`). The only source call site inside `wait_for_completion` calls `poll_backend_completion(tab, conv, baseline)` without `prefer_lightweight=True` (`completion.py:162`), and `Session._run_send_turn` calls `wait_for_completion` without changing that (`src/ask_chatgpt/session.py:444-452`). Therefore the live default completion poll is a full `GET /backend-api/conversation/<id>`, not `/stream_status`.

The default interval is 30 s. `wait_for_completion` defaults `progress_poll_interval_s=2.0` (`src/ask_chatgpt/completion.py:135`), computes `backend_interval = _backend_interval(...)` (`completion.py:140`), initializes `last_backend_check = start - backend_interval` so the first backend check is immediate (`completion.py:141`), and `_backend_interval` returns `max(progress_poll_interval_s * 2, 30.0)` when no override is supplied (`completion.py:470-473`). With the default, that is `max(4, 30)=30` seconds. A response that keeps the wait loop active for `R` seconds therefore executes backend polls at approximately `t=0,30,60,...`; exact long-run count is about `1 + floor(R/30)` if not completed by DOM first, and the issue’s rough rate `R/30` is fine for long responses. For 40 min, `2400/30=80` intervals; including the immediate first poll, the code can execute about 81 full-conversation GETs.

## One-call OLD-path baselines

Definitions: let `R` be seconds spent in `wait_for_completion`; let `P(R) ≈ 1 + floor(R/30)` be the default backend completion-poll count while the wait loop remains active; let `B_render` and `B_light` be the not-statically-enumerable `/backend-api/*` boot burst caused by a render `/c/<id>` navigation and light `https://chatgpt.com/` navigation respectively.

| Persistent `loop` over `N` existing-conversation turns | `loop` acquires once, but every `_run_send_turn` still reloads before send | `1` initial render navigation + `N` pre-send reload navigations + `N * (SPA send + P(R_i) completion full GETs + 1 capture full GET)` |

The persistent-tab `loop` does not currently achieve “one page-load per session.” It eliminates per-iteration `TabPool.acquire` opens (`src/ask_chatgpt/session.py:563-606`, acquire at `session.py:583`), but `_run_send_turn` still calls `wait_for_idle_and_reload_if_needed` on every iteration (`session.py:404`), and that function reloads once idle (`src/ask_chatgpt/send.py:54-56`). Thus a persistent `loop` currently has approximately one full page reload per turn, plus the initial open.

## Driver pattern confirmation

The gitignored driver does spawn new CLI processes: `run_scrape` invokes `ask-chatgpt scrape` (`tmp/weak-simplex-push/driver/driver.sh:106`), `wait_until_idle` calls `run_scrape` in its loop (`driver.sh:134-151`), `run_ask` invokes `ask-chatgpt ask` (`driver.sh:181-189`), the main loop calls `run_ask` per turn (`driver.sh:223-225`), and it performs a confirmation `run_scrape` after a provisional send count increase (`driver.sh:231-239`). The idle interval is config-driven: default loader uses `.idle_poll_interval_s // 30` (`driver.sh:61`), and the current config line I inspected sets it to 300 s (`tmp/weak-simplex-push/driver/driver.config.json:30`). Therefore each driver scrape poll adds the base scrape profile above: one light page-load burst plus one full conversation GET.

## Dominant non-human signal, with numbers

For a single long `ask` on an existing conversation, the largest statically countable explicit backend volume is the completion poll: about `R/30` full `GET /backend-api/conversation/<id>` calls, with an immediate first check making a 40-min response about 81 polls, plus 1 capture GET. The page-navigation count for that same single `ask` is 2 render navigations (open + pre-send reload), or 3 for a draft. Therefore, by raw explicit `/backend-api/conversation` request count on a long response, the 30 s poll dominates.

For human-likeness/fingerprint, repeated page-loads remain a critical dominant signal because humans keep one tab open, while the old driver pattern creates fresh `ask`/`scrape` processes and the tool also reloads before every send. A current 40-min idle scrape gate at 300 s adds roughly `2400/300 = 8` scrape polls, hence about 8 light page-load bursts and 8 full conversation GETs; at a historical 90 s interval it would be about 27. Those are in addition to any ask-side completion polls. The key correction is that both sources matter: page-load churn is the most visibly non-human browser pattern, but the completion loop is the numerically heavier explicit backend fetcher for long waits, and it is polling the full conversation endpoint rather than `/stream_status`.
