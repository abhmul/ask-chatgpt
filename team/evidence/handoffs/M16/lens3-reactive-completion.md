## 3. Mechanism verdict

The correct implementation direction is **not** to reuse `StreamState` for the SPA's native send stream. Build a new Network lifecycle observer on top of the already-wired CDP session in `CdpChannel._install_cdp_observers` (`src/ask_chatgpt/channels/cdp.py:889-902`): add listeners for `Network.responseReceived`, `Network.loadingFinished`, `Network.loadingFailed`, and optionally `Network.eventSourceMessageReceived`; persist redacted lifecycle events keyed by CDP `requestId`; expose a `wait_for_response_finished`/`wait_for_native_stream_close` method through the channel seam or a CDP-specific capability. Then `_run_send_turn` can arm the observer before `submit_composer` (`src/ask_chatgpt/session.py:418`) and use stream-close as the primary completion signal before final capture (`src/ask_chatgpt/session.py:457`).

This is **new code on existing Network infra**, not a browser/control re-architecture. The remaining uncertainty is semantic/live-site: whether the particular ChatGPT response stream emits `loadingFinished` exactly when the assistant turn is complete.

## 4. Minimal confirming probe spec (design only; do not run in this mission)

Single attended leg, one short non-sensitive send in an expendable conversation; record only methods, URL paths, status codes, event names, request IDs, and monotonic timings. Do **not** record prompt text, response text, header values, cookies, auth tokens, file IDs, or bodies.

1. Attach to the attended browser CDP session and enable `Network.enable` on the page's CDP session.
2. Start recording these CDP events: `Network.requestWillBeSent` (requestId, method, path, timestamp), `Network.responseReceived` (requestId, status, resource/type, content-type header name only if needed), `Network.loadingFinished` (requestId, timestamp), `Network.loadingFailed` (requestId, errorText category only), and `Network.eventSourceMessageReceived` (requestId, event name/timestamp only; omit message data). Also record Playwright `requestfinished` for cross-check.
3. Immediately before the send-button click, mark `submit_click_t0`. Identify candidate native send-stream requests as `/backend-api/*` requests starting after `t0`, especially a POST or streaming/event-stream response associated with the conversation.
4. Confirm the candidate request ID receives `loadingFinished`/`requestfinished` after the final assistant response is visible and the UI is idle, not before. Continue only a short guard window (e.g. 3-5 s) to confirm no further assistant DOM text changes after the finish event.
5. Pass criterion for feasibility: exactly one identified native response stream for the send has a close event whose timing matches the turn finishing; no `/backend-api/conversation/<id>` polling request is needed to decide completion. If the stream is SSE and `eventSourceMessageReceived` fires, its messages may corroborate liveness, but message payloads must not be recorded.

## 5. Robustness vs today's heuristic and fallback recommendation

Today's code heuristic is backend polling plus DOM stability. The DOM side polls `query_turns` (`src/ask_chatgpt/completion.py:173`), can complete only when a new assistant id exists, stop is absent, and text is non-empty (`src/ask_chatgpt/completion.py:401-405`), and then requires the same assistant id/text-hash/length for `stable_window_s` (`src/ask_chatgpt/completion.py:178-190`, key at `:410`). In code, `stable_window_s` equals `progress_poll_interval_s` (`src/ask_chatgpt/completion.py:147`), default 2.0 s (`:135`); I did not find a separate hard-coded 3 s/global-marker path in `completion.py`.

A native stream-close signal should be **more robust for short multi-turn responses** than stop-button/text-stability alone, because it ties completion to the browser's actual server response ending instead of inferring from transient DOM markers. It is **less robust as a sole signal** if the SPA uses reconnects, multiplexing, early stream close before final DOM flush, request failures, or event types other than fetch/SSE. Recommendation: make stream-close the primary path to eliminate backend polling, but keep a DOM/heuristic fallback/guard: after stream close, require a brief DOM-stability confirmation before capture; if no candidate stream is observed or it fails/ambiguous, fall back to the current DOM heuristic and only use backend polling as an explicit low-rate rescue, not the default live path.
