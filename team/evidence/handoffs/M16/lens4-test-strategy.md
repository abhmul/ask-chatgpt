## Ground-truth corrections that drive the test design

The current completion backend poll is a full `GET /backend-api/conversation/<conversation-id>`, not `/stream_status`, on the live default path: `poll_backend_completion` chooses `/stream_status` only when `prefer_lightweight=True`, otherwise the full conversation path (`src/ask_chatgpt/completion.py:37`, `src/ask_chatgpt/completion.py:46`), and `wait_for_completion` calls it without that flag (`src/ask_chatgpt/completion.py:159`, `src/ask_chatgpt/completion.py:162`). The default cadence is immediate plus every 30s: `progress_poll_interval_s` defaults to 2s (`src/ask_chatgpt/completion.py:135`), `_backend_interval` returns `max(progress_poll_interval_s * 2, 30.0)` (`src/ask_chatgpt/completion.py:470`, `src/ask_chatgpt/completion.py:473`), and `last_backend_check = start - backend_interval` causes an immediate first check (`src/ask_chatgpt/completion.py:141`). Formula when DOM does not end the wait first: `backend_full_gets(T) = 1 + floor(T / 30s)`, so a 40-minute response is 81 full conversation GETs before any post-completion capture GET.

The page-load test must count reloads, not just tab opens. `Session.loop` acquires once (`src/ask_chatgpt/session.py:563`, `src/ask_chatgpt/session.py:583`), so an open-tab-only assertion can pass on current main. But every send goes through `_run_send_turn`, which calls `wait_for_idle_and_reload_if_needed` (`src/ask_chatgpt/session.py:404`), and that helper reloads whenever the stop button is absent (`src/ask_chatgpt/send.py:48`, `src/ask_chatgpt/send.py:54`, `src/ask_chatgpt/send.py:55`). `MockChannel` already records both `open_tab` and `reload` (`src/ask_chatgpt/channels/mock.py:285`, `src/ask_chatgpt/channels/mock.py:293`, `src/ask_chatgpt/channels/mock.py:302`, `src/ask_chatgpt/channels/mock.py:311`).

There is no current 429/`Retry-After` branch in the inspected paths. Completion non-200 raises `BackendCaptureShapeError` (`src/ask_chatgpt/completion.py:54`, `src/ask_chatgpt/completion.py:58`); capture non-2xx raises `BackendCaptureShapeError` (`src/ask_chatgpt/capture.py:937`, `src/ask_chatgpt/capture.py:938`), then `capture_conversation` falls into UI fallback (`src/ask_chatgpt/capture.py:358`, `src/ask_chatgpt/capture.py:360`), whose default path raises `HumanActionNeededError` for clipboard permission (`src/ask_chatgpt/capture.py:531`, `src/ask_chatgpt/capture.py:532`). Existing exit codes include 21 for `HumanActionNeededError` and 41 for `BackendCaptureShapeError` (`src/ask_chatgpt/errors.py:142`, `src/ask_chatgpt/errors.py:144`, `src/ask_chatgpt/errors.py:198`, `src/ask_chatgpt/errors.py:200`), but no distinct rate-limit error class is present in the taxonomy around those definitions.

## Layer 2: request-counting harness for one short attended tool leg

Build a passive CDP Network observer, separate from send/capture code, that attaches to the already selected tool-owned tab and records only request metadata. Current `CdpChannel` already installs page request observers and CDP request observers (`src/ask_chatgpt/channels/cdp.py:878`, `src/ask_chatgpt/channels/cdp.py:885`, `src/ask_chatgpt/channels/cdp.py:889`, `src/ask_chatgpt/channels/cdp.py:900`, `src/ask_chatgpt/channels/cdp.py:902`), but the counting harness should also subscribe to `Network.responseReceived`, `Network.loadingFinished`, `Network.loadingFailed`, and optionally `Page.frameNavigated` because current production code only records request starts/extra headers and leaves `requestfinished` as a no-op (`src/ask_chatgpt/channels/cdp.py:917`).

Record per event: monotonic timestamp relative to harness start, event name, requestId hash or sequence number, HTTP method, normalized path with dynamic file ids redacted, status code for `responseReceived`, resource type, and whether it is a top-level document/navigation. Never record header values, cookies, authorization/OAI values, request/response bodies, post data, conversation text, file ids, or attachment bytes. Header names are not needed for this harness.

Pass criterion for a 3-turn attended tool leg: at most one initial page-load burst/top-level document navigation for the session; zero reload/document navigations after the initial burst; zero `/backend-api/conversation/<id>/stream_status`; zero periodic `GET /backend-api/conversation/<id>` inside each send-to-stream-close generation window; and at most one post-stream `GET /backend-api/conversation/<id>` per turn if capture-from-stream is not implemented. The structural comparison is against the old formula above: old completion polling alone is `sum_i (1 + floor(T_i / 30s))` full conversation GETs for response durations `T_i`, whereas the fixed path should have zero completion GETs and only optional capture GETs.

The harness observes only. It must not call `fetch_in_page`, `wait_for_request` in a way that waits for missing requests by causing reloads, browser `reload`, `curl`, `ask-chatgpt`, or any send action. The operator/tool under test produces the normal turns; the observer just counts events.

## Layer 3: human-baseline capture

Use the same passive observer for one attended, operator-driven human message in the ChatGPT SPA. Start with the tab already loaded and idle; record the optional one page-load burst separately from the per-turn send profile. The operator manually sends one short message; the harness records paths/counts/statuses/event names only. This defines the “looks human” target: no tool-side completion GET loop, no per-turn reload, and whatever SPA-native send/stream path and close event the browser naturally emits. Do not record message text, request payloads, response bodies, or header values.

## Layer 4: what genuinely needs real-site exposure

Offline tests plus governor math can prove the code structure: no backend completion fetches in the mocked live path, one open tab with no per-turn reloads, enforced spacing from a fake clock, and correct 429/`Retry-After` handling. They cannot prove the server/account will never rate-limit a long run, because that ceiling is opaque and shared with human/operator activity. Therefore a long unattended soak should not be a CI or repeated verification method.

Minimum real-site exposure I would require before rollout: one human-baseline message and one short attended instrumented tool leg of about three turns, both with the passive observer and no extra scrapes. If the operator insists on directly falsifying the “hours, many turns” criterion before deploy, the minimum is a single bounded attended pilot under the new governor, with a hard stop on any 429/modal, no supervision scrapes, no repeated soak, and all monitoring from local files plus the passive event counter. The pilot is not a proof of future absence; it is only a final smoke/falsification gate.

## Explicitly forbidden test designs

Do not test by polling chatgpt.com from the harness, scraping every N seconds, repeatedly reloading pages to count requests, running an 8-hour soak to “see if it happens,” or using the installed `ask-chatgpt` binary against the live site as a test generator. Any real-site leg must be attended, bounded, operator-approved, and observed passively; all bulk proof must be offline with `MockChannel`.
