## Findings

1. **MUST-FIX before M4 implementation: the channel/tab seam is not implementable exactly as typed.** Evidence: the V3 contract requires checking that "every module has concrete typed signatures" (`team/contracts/M3-verify-3-occam-impl.md`, "What to check" item 2). The design says lower modules should use the browser protocol: "Production modules call only this Protocol, never Playwright directly" and defines `BrowserChannel.wait_for_request(self, tab: BrowserTab, ...)`, `fetch_in_page(self, tab: BrowserTab, ...)`, and `query_turns(self, tab: BrowserTab, ...)` (`team/evidence/reports/M3-detailed-design.md`, §2.9). But the load-bearing module signatures omit the channel and pass only a thin tab record: `BrowserTab` has only `tab_id` and `url` (§2.9), while `capture.py` exposes `def acquire_backend_headers(tab: BrowserTab, ...)`, `def stream_backend_conversation(tab: BrowserTab, ...)` (§2.3), `send.py` exposes `def send_prompt(tab: BrowserTab, ...)` (§2.4), and `completion.py` exposes `def wait_for_completion(tab: BrowserTab, ...)` (§2.5). As written, an implementer must invent whether `BrowserTab` secretly owns a channel, whether a global channel is used, or whether all signatures need `channel: BrowserChannel`. This is load-bearing because header capture, backend fetch, baseline reads, and DOM polling all require channel calls. Fix: define a `TabLease`/`BrowserTab` object that is explicitly channel-bound, or add `channel: BrowserChannel` to capture/send/completion/menu signatures; also define the referenced but unspecified seam types (`Transcript`, `SelectorMap`, `SendTimeouts`, `AttachmentSpec`, `PreflightResult`, `StatusReport`) in the design contract.

2. **MUST-FIX before real completion work: backend polling has no empirically bounded request-cost plan and may accidentally retain auth headers longer than the one-fetch rule.** Evidence: M2 measured the confirmed conversation endpoint at "about 17.1 MB in one response" with "No pagination" and about "5.0k mapping nodes" (`team/evidence/handoffs/M2-ground-truth-probe.md`, "Backend-api verdict"). The design sets `poll_interval_s: float = 2.0` in `wait_for_completion` (§2.5) and says the "Primary loop polls authenticated backend conversation state" (§5), while only adding that "Completion polling must not rewrite a 17 MB raw file on every poll" (§4.2). That prevents disk churn but does not bound request count, CDP/page-binding transfer, browser memory, or parse cost for minutes-long Pro/DR runs. Agent rigor requires that "Any claim about time, space, disk, memory, request-count, or scale MUST be empirically grounded" (`.claude/skills/manager/references/agent-rigor.md`, "Measure complexity empirically"). In addition, the common constraint says auth/OAI headers must be obtained transiently, "forward them for the single fetch, and discard" (`team/contracts/M3-common-constraints.md`, §2), but the design stores `_headers` in `HeaderBundle` (§2.3) and `poll_backend_completion(..., headers: HeaderBundle | None = None)` (§2.5) is ambiguous about reuse across polls. Fix: specify a concrete completion cadence that is empirically measurable, e.g. coarse full-backend polls plus DOM progress between polls until `stream_status` is verified, and make `HeaderBundle` one-shot/consumed per fetch or explicitly reacquired per poll without persistence/logging.

3. **NICE-TO-HAVE / sequencing correction: the recommended M4 scope accretes M7 work and can delay the pressing scrape path.** Evidence: the authoritative mission sequence is `M4` offline core, `M5` backend-api capture + scrape + verified send, `M6` run the target scrape + verify fidelity, then `M7` "model/tools + keep-pushing loop + tab-pool/rate" (`docs/REWRITE-SPEC.md`, §19). The design's M4 sequence already includes `menus.py` (§10 M4 step 6) and "Implement `TabPool` and `AdaptiveSendBudget`" (§10 M4 step 7), and its M5 step 3 says to "Run `scrape` of the authorized target conversation" even though §19 names that as M6. This is not an implementation blocker, but it is non-Occam for the build plan: a minimal one-tab session and store/capture spine is enough to reach M6; full tab-pool/rate/menu/loop polish can remain M7 unless needed by a falsifiable mock test.

4. **NICE-TO-HAVE before model/tool implementation: the model-picker selector is still prose, not an executable selector/algorithm.** Evidence: M2 observed only a heuristic, "button[aria-haspopup=\"menu\"] with visible current-model label in the composer footer; no stable data-testid/aria-label observed" (`team/evidence/handoffs/M2-ground-truth-probe.md`, "Selectors"). The design carries this into `selectors/real.json` as `"model_picker_heuristic": "composer-footer button[aria-haspopup=\"menu\"] showing the current-model label; no stable data-testid/aria-label"` (§2.10), which is not CSS and is not a fully specified DOM scan. This can remain fail-closed, but before M7 it should say exactly how to enumerate candidate buttons, how to disambiguate multiple matches, and what reflected label proves success.

5. **PASS: the main Occam choices are sound.** Evidence: the design keeps the approved architecture, "library-core + thin CLI + persistent Session; no daemon" (§1), matching `REWRITE-SPEC.md` §2, and explicitly cuts the measured-overlarge tab cache: "Lens 4 proposed `pool_max_tabs=6` is cut as unmeasured over-capacity" (§7). It also rejects the old over-engineered bundle/patch/apply path: `REWRITE-SPEC.md` §15 says the v1 bundle/patch/apply round-trip is "Dropped (over-engineered)", and the design replaces it with general attachment refs without resurrecting patch/apply (§3.7). The single shared `TurnRecord` seam (§2.1) is a simplification, not an abstraction tax.

6. **PASS: the core transcript and linearization design is mostly implementable.** Evidence: the design specifies a JSONL record table with required fields, nullability, statuses, read semantics, supersession, and atomic write behavior (§3.3–§3.4). The current-branch algorithm is explicit pseudocode following parent links from `current_node` to root and reversing the path (§3.5), grounded in M2's fact that `mapping` is a message tree (`M2-ground-truth-probe.md`, "Backend-api verdict"). The visible/hidden rule is also concrete: emit `user:text` and `assistant:text`; keep `assistant:code`, thoughts, recap, model-editable-context, tools, and system raw-only (§3.5), matching M2's role/content observations.

7. **PASS: the core acceptance tests are falsifiable, not purely self-reported.** Evidence: the design's M5 fidelity acceptance says to "compare backend/copy/fallback output to web-UI copy" and require `\widehat`, `\ne`/`\neq`, and `\frac{}{}` to round-trip (§4.4), matching `REWRITE-SPEC.md` §5 and §18's requirement that tests be checkable against an authoritative signal. Send/completion tests are also falsifiable: the design requires a pre-send baseline and a new user turn carrying the prompt (§6), and M4 mock acceptance includes "no-op send raises `PromptNotSubmittedError`" and "completion requires newer assistant id" (§10 M4 step 5). The mock is rich but not vacuous: it must simulate 404-without-headers, required-header success, no-op send, composer unmount, long progress, selector drift, and clipboard prompt (§2.9, §10 M4 step 3).

## Prioritized must-fix vs nice-to-have

**Must-fix:** (1) make the `BrowserChannel`/`BrowserTab`/module-call seam concrete and define all referenced types; (2) define an empirically bounded completion polling strategy and one-shot header lifetime semantics.

**Nice-to-have:** (1) trim/postpone M4 work that belongs to M7 so M6 scrape remains the shortest path; (2) replace `model_picker_heuristic` prose with a fail-closed executable DOM algorithm.

**Single biggest risk to implementability:** the missing channel-bound tab seam, because it affects every load-bearing real/mocked browser operation: header acquisition, backend fetch, send baseline, prompt verification, completion polling, menus, and fallback capture.

Final verdict: **PASS-WITH-NOTES**.
