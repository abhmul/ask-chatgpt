# M4 test-plan Lens 1: offline core data model, identity, allowlist, selectors, store

Sources read: `team/contracts/M4-common.md`, `team/contracts/M4-offline-core.md`, and `team/evidence/reports/M3-detailed-design.md` §§2.1, 2.7, 2.8, 2.10, 2.11, 3, 9, plus M4 build-sequence acceptance lines. This is a falsifiable behavior checklist, not test code. Scope deliberately excludes capture parsing, send/completion mechanics, and real CDP/browser behavior except where `history`/`export` must prove they do not preflight CDP.

## A. Public data model and error contracts

1. **TurnRecord is the single persisted/returned turn seam.** Behavior: public construction/serialization of a turn uses one `TurnRecord` shape with all fields named in M3 §2.1/§3.3; there is no separate persisted `AskResult` schema with divergent fields. Why it matters: prevents drift between `capture.py`, `Session.ask`, and JSONL, which is an M4 Step 1/2 acceptance premise. Falsifiability: an implementation that persists only `role/content/message_id` or returns an assistant wrapper without `attachments`, `citations`, `status`, `partial`, `client_send_id`, etc. must fail. Required mock/fixture/setup: a complete assistant `TurnRecord` fixture containing model, tools, attachment, citation, exchange id, and error-null fields.

2. **ModelRef round-trips nullable slug/display.** Behavior: `ModelRef(slug=None, display=None)` and a populated model serialize as JSON object/null exactly where allowed and load back without inventing values. Why it matters: model is independent of tools and may be unavailable offline; tests must not force live-site guesses. Falsifiability: an implementation that converts missing slug to `""`, drops display, or stores model labels in `active_tools` fails. Required mock/fixture/setup: turns with `model=None`, `ModelRef(None, "Pro Extended")`, and `ModelRef("gpt-4", None)`.

3. **AttachmentRef preserves all store-level fields without fetching bytes.** Behavior: JSONL round-trip preserves `source_kind`, `source_ref`, `raw_path`, filename/mime/bytes/sha/local path/download state, and nested sanitized metadata; no lazy network/file fetch is triggered by serialization or render. Why it matters: attachments are references in M4; bytes are lazy and capture normalization is Lens 2. Falsifiability: an implementation that serializes only filenames, turns citations into attachments, or attempts to open/download `local_path` during load/render fails. Required mock/fixture/setup: one turn with four representative `AttachmentRef` values (`user_upload`, `file_reference`, `generated_asset`, `code_execution_output`) and a fake downloader/open hook that would fail if called.

4. **CitationRef preserves citation metadata and remains non-downloadable.** Behavior: JSONL round-trip preserves title, url, source, type, offsets, citation format, raw path, and metadata, and store/render never fetches citation URLs. Why it matters: citations are source references, not attachments; fetching them would violate the design and potentially leak network access. Falsifiability: an implementation that writes citations into `attachments/`, fetches citation URLs, or drops offsets fails. Required mock/fixture/setup: assistant turn with at least two citations, including one URL on a non-allowlisted host to prove render/load does not fetch it.

5. **`partial` and `status` invariant is enforced at the public boundary.** Behavior: accepted records satisfy `partial is False` iff `status == "complete"`; pending/error/partial records have `partial=True`. Invalid combinations are rejected before durable persistence or normalized in one documented public constructor path. Why it matters: consumers use `partial` as a redundant convenience, so contradictory records make recovery unsafe. Falsifiability: an implementation that writes `status="complete", partial=True` or `status="error", partial=False` to JSONL and then treats it as valid fails. Required mock/fixture/setup: valid complete/partial/error records plus two invalid records.

6. **Only pending eager-write stubs may use local message ids.** Behavior: a `message_id` beginning `local:` is accepted only for a pending user stub with `turn_index=None`, `created_at=None`, `status="partial"`, `partial=True`, and a `client_send_id`; backend-derived complete records use canonical ids. Why it matters: prevents unsubmitted prompts from appearing as real ChatGPT turns. Falsifiability: an implementation that permits `local:abc` on a complete assistant record or assigns a numeric turn index to a stub fails. Required mock/fixture/setup: one valid `begin_send` stub and invalid local-id variants.

7. **Dataclass immutability is observable.** Behavior: public dataclasses declared frozen in the design cannot be mutated after construction; updates are represented by appending replacement records. Why it matters: append-only/last-writer-wins relies on immutable records rather than in-place mutation. Falsifiability: an implementation where `record.content_markdown = "changed"` mutates an existing object silently fails. Required mock/fixture/setup: a minimal `TurnRecord`, `ModelRef`, `AttachmentRef`, and `CitationRef` instance.

8. **Named errors expose stable codes, exit codes, retryability, and redacted details.** Behavior: all listed errors inherit `AskChatGPTError` and expose the code/exit-code table in M3 §9, including `DomainNotAllowedError` exit 22, `SelectorNotFoundError` exit 24, `StoreError` exit 70, and `InternalError` exit 99. Why it matters: CLI/status behavior is stable and tests can assert safety failures without matching prose. Falsifiability: an implementation using generic `ValueError`, wrong exit code, or details containing cookies/OAI/bearer headers fails. Required mock/fixture/setup: instantiate representative errors with sensitive-looking details and inspect public attributes/string/JSON rendering.

## B. Identity parsing and conversation references

9. **Bare conversation ids resolve statelessly.** Behavior: `parse_conversation_address` accepts a bare conversation id, returns `ConversationRef(conversation_id=<id>, project_id=None, url=https://chatgpt.com/c/<id>)`, and does not require `index.json`. Why it matters: URL/id addressing must work without a prior registry. Falsifiability: an implementation that treats bare ids only as aliases or requires an index entry fails. Required mock/fixture/setup: empty temp data dir and a representative id such as `6a316aa8-5dc8-83ea-9014-b8ea38dabc31`.

10. **Plain chat URLs are parsed.** Behavior: `https://chatgpt.com/c/<conversation_id>` parses to the same canonical id with `project_id=None`; query strings/fragments do not become part of the id. Why it matters: this is one of the two lead-required URL shapes. Falsifiability: an implementation that includes `?model=...` in the id or rejects a valid `/c/<id>` URL fails. Required mock/fixture/setup: plain URL with and without trailing slash/query/fragment.

11. **Project chat URLs are parsed.** Behavior: `https://chatgpt.com/g/g-p-<projid>/c/<chatid>` parses to `conversation_id=<chatid>`, captures the project id consistently, and canonicalizes `conversation_url` back to the project URL shape. Why it matters: lead decision makes both URL shapes in M4 scope, while project send/create remains out of this lens. Falsifiability: an implementation that returns the project id as the conversation id, drops project metadata, or canonicalizes to non-project `/c/<chatid>` despite knowing the project fails. Required mock/fixture/setup: project URL fixture such as `https://chatgpt.com/g/g-p-proj123/c/chat456`.

12. **`parse_project_address` recognizes project addresses without inventing a conversation.** Behavior: a project URL or project path returns the project id, while a plain conversation URL returns `None`; it must not create a draft conversation ref. Why it matters: project metadata is addressable, but project create/send is outside M4. Falsifiability: an implementation that treats `/g/g-p-x` as conversation id `x` or returns a draft `ConversationRef` from `parse_project_address` fails. Required mock/fixture/setup: project root URL, project chat URL, plain chat URL, malformed project URL.

13. **Conversation URL helpers are canonical and reversible.** Behavior: `conversation_url(ref)` emits `/c/<id>` when `project_id` is absent and `/g/g-p-<project_id>/c/<id>` when present; `backend_conversation_url(id)` uses only the conversation id backend key; parsing the emitted URL returns an equivalent ref. Why it matters: store paths are keyed by chat id while web reopening may need project context. Falsifiability: an implementation that includes project id in the backend URL or loses project URL shape on round-trip fails. Required mock/fixture/setup: one plain `ConversationRef` and one project `ConversationRef`.

14. **Malformed or foreign conversation addresses fail closed.** Behavior: non-chatgpt hosts, wrong paths, empty ids, and path traversal-like ids return `None` from parse helpers or raise the documented public error from resolver, without falling back to a guessed id. Why it matters: prevents accidental navigation/fetch outside the intended domain and avoids corrupt store paths. Falsifiability: an implementation that accepts `https://evil.example/c/abc`, `/c/../../x`, or `https://chatgpt.com/not-c/abc` fails. Required mock/fixture/setup: malformed address table.

15. **Alias/session resolution is optional and layered after stateless parsing.** Behavior: `resolve_conv_or_alias` returns a passed `ConversationRef` unchanged, parses URLs/bare ids without index dependence, and resolves aliases/sessions from `index.json` only when parsing does not produce an address. Why it matters: aliases are convenience cache, not the primary identity mechanism. Falsifiability: an implementation that lets an alias named like a URL override the URL, or cannot resolve a URL when index is corrupt, fails. Required mock/fixture/setup: index with `aliases` and `sessions`, plus an empty/corrupt index case.

16. **Missing aliases fail with `ConversationNotFoundError`.** Behavior: resolving an unknown non-address alias raises the stable `CONVERSATION_NOT_FOUND` error rather than creating a new id or returning `None` from a resolver that promises a ref. Why it matters: silent creation would route prompts/history to the wrong conversation. Falsifiability: an implementation that fabricates `ConversationRef(conversation_id="missing")` for an unknown alias fails. Required mock/fixture/setup: temp store with no alias named `missing`.

## C. Allowlist URL validation

17. **Default allowlist accepts every documented suffix and apex.** Behavior: `Allowlist().is_allowed_url` returns true for `chatgpt.com`, subdomains of `chatgpt.com`, `openai.com`, subdomains of `openai.com`, `oaiusercontent.com`, subdomains of `oaiusercontent.com`, `oaistatic.com`, and subdomains of `oaistatic.com`. Why it matters: these are the exact `DEFAULT_ALLOWED_HOST_SUFFIXES` required by M3 §2.11. Falsifiability: an implementation that accepts only `chatgpt.com` and rejects `auth.openai.com`/`files.oaiusercontent.com`, or accepts only subdomains but rejects apexes, fails. Required mock/fixture/setup: URL table covering all eight suffix entries.

18. **Default allowlist rejects suffix-confusion and unrelated hosts.** Behavior: hosts such as `evilchatgpt.com`, `chatgpt.com.evil.example`, `openai.com.attacker.test`, and `example.com` are rejected. Why it matters: naive substring matching is unsafe for navigation/fetch. Falsifiability: an implementation using `"chatgpt.com" in host` fails. Required mock/fixture/setup: malicious host table with both apex-looking and subdomain-looking strings.

19. **Allowlist rejects non-URL or unsafe-scheme inputs.** Behavior: `require_allowed_url` rejects empty strings, relative paths, `file:`, `javascript:`, `data:`, and other non-http(s) navigation/fetch targets even if their text contains an allowed host. Why it matters: allowlist is a navigation/fetch safety boundary, not just a hostname helper. Falsifiability: an implementation that parses `javascript:https://chatgpt.com` or `file://chatgpt.com/tmp` as allowed fails. Required mock/fixture/setup: scheme/input table.

20. **`require_allowed_url` raises a stable redacted `DomainNotAllowedError`.** Behavior: rejected URLs raise `DomainNotAllowedError` with code `DOMAIN_NOT_ALLOWED`, exit code 22, and redacted/sanitized URL details. Why it matters: CLI/status can map safety failures deterministically without leaking credentials. Falsifiability: an implementation that raises `AssertionError`, returns false silently from `require_allowed_url`, or includes `?access_token=secret` in its message fails. Required mock/fixture/setup: disallowed URL containing userinfo, query token, and fragment.

21. **`sanitize_for_log` keeps enough host context but redacts secrets.** Behavior: sanitized output identifies the host/class of URL while removing credentials, query strings, fragments, bearer-looking tokens, and cookies. Why it matters: errors must be actionable and safe. Falsifiability: an implementation that returns the original URL with `user:pass@host/path?token=...` fails; an implementation that returns only `"<redacted>"` for every URL is also too weak if it hides the domain needed for debugging. Required mock/fixture/setup: allowed and disallowed URLs containing credentials/query/fragment.

## D. Selector-map schema validation

22. **The real selector map contains every required M2 key.** Behavior: loading the packaged selector map in strict mode yields all required keys: `composer`, `tools_button`, `message_turn`, `user_turn`, `assistant_turn`, `copy_button`, `stop_button`, `send_button_unverified_no_input`, `radix_portal`, and `model_picker_trigger_candidates`. Why it matters: M4 Step 1 acceptance explicitly requires selector schema validation and the Radix/model-picker additions. Falsifiability: an implementation that ships a pre-M2 selector file missing `radix_portal` or `model_picker_trigger_candidates` fails. Required mock/fixture/setup: packaged `selectors/real.json` plus a strict loader/validator entry point.

23. **Strict validation fails when any required key is missing.** Behavior: a selector map missing each required key, one at a time, fails startup/config validation with `SelectorNotFoundError`/`SELECTOR_NOT_FOUND`. Why it matters: selector drift must fail closed before browser actions. Falsifiability: an implementation that only validates `composer` and `send_button` or fills missing keys with `""` fails. Required mock/fixture/setup: table-generated selector-map fixtures with one omitted key per case.

24. **Strict validation rejects non-string and empty selector values.** Behavior: required selector values must be non-empty strings; `null`, arrays, booleans, and whitespace-only strings fail. Why it matters: a key can be present but unusable, causing later misleading send/capture failures. Falsifiability: an implementation that checks only key presence fails. Required mock/fixture/setup: selector maps with invalid value types for representative keys.

25. **Extra selector keys do not compensate for missing required keys.** Behavior: forward-compatible extra keys may be preserved/ignored, but they cannot substitute for exact required names. Why it matters: prevents typoed schema (`model_picker_trigger_candidate`) from passing. Falsifiability: an implementation that accepts a map with `model_picker_trigger_candidate` but no `model_picker_trigger_candidates` fails. Required mock/fixture/setup: selector map with typo plus otherwise valid values.

26. **Model-picker selector is treated as candidates, not a trusted single selector.** Behavior: the schema stores `model_picker_trigger_candidates` as the candidate-query string named in M3 and validation does not rename it to a final model-picker selector. Why it matters: M2 found no stable test id/aria label; later menu code must enumerate and match visible labels. Falsifiability: an implementation that exposes only `model_picker_trigger` or drops the candidates key passes naive tests but must fail here. Required mock/fixture/setup: packaged selector map and public selector map type/loader inspection.

## E. Store layout, data-dir resolution, and index

27. **Data-dir resolution follows the documented priority.** Behavior: explicit `Store(data_dir=...)`/CLI `--data-dir` wins over `ASK_CHATGPT_DATA_DIR`, which wins over `~/.local/state/ask-chatgpt/`; resolved paths are expanded consistently. Why it matters: agents need isolated temp stores and predictable persistence. Falsifiability: an implementation that always uses the env var or current directory fails. Required mock/fixture/setup: temp explicit dir, env mapping with a different dir, and env-empty case with home patched or asserted via path suffix.

28. **Conversation layout is created exactly and privately enough.** Behavior: `ensure_conversation(ref)` creates `<data-dir>/conversations/<conversation-id>/transcript.jsonl`, `raw-mapping.json` path parent, `attachments/`, and `.gitignore` containing `attachments/`, without deleting existing files. Why it matters: transcript/raw/attachments separation and lazy attachment cache are part of the M4 store contract. Falsifiability: an implementation that stores files directly under data dir, omits `.gitignore`, or overwrites an existing transcript fails. Required mock/fixture/setup: temp data dir with preexisting sentinel file and a `ConversationRef`.

29. **Conversation paths are keyed by conversation id, not project id or alias.** Behavior: project conversations still use `conversations/<chatid>/` as the directory; aliases/sessions never create alternate transcript copies. Why it matters: backend API and JSONL key are the chat id in both URL shapes. Falsifiability: an implementation that stores project chat under `conversations/<projectid>/` or `conversations/<alias>/` fails. Required mock/fixture/setup: project `ConversationRef` and alias in index pointing to it.

30. **`put_conversation_ref` creates/patches `index.json` without clobbering unrelated state.** Behavior: index has `schema_version: 1`, `aliases`, `sessions`, and `conversations`; updating one conversation preserves existing aliases/sessions and other conversation entries. Why it matters: index is a convenience cache and must not destroy user aliases. Falsifiability: an implementation that rewrites index to contain only the latest conversation fails. Required mock/fixture/setup: preloaded index with alias/session and two refs.

31. **Index metadata comes from refs/backend data, not agent wall clock.** Behavior: stored `last_updated` is derived from `ConversationRef.updated_at` when present and remains null/absent when unavailable; store does not invent current time. Why it matters: M3 explicitly rejects agent wall-clock timestamps for backend-derived metadata. Falsifiability: an implementation that sets `last_updated` to `datetime.now()` for every update fails under a frozen-clock/sentinel assertion. Required mock/fixture/setup: refs with `updated_at=None` and a fixed aware datetime.

32. **Corrupt index is non-fatal for stateless addresses.** Behavior: if `index.json` is invalid JSON, resolving a bare id or URL and loading an existing conversation still works; alias lookup may fail gracefully/rebuild rather than crash unrelated reads. Why it matters: M3 says index corruption is non-fatal and rebuildable. Falsifiability: an implementation that refuses all store operations when `index.json` is corrupt fails. Required mock/fixture/setup: temp store with valid conversation directory/transcript and deliberately corrupt index text.

## F. JSONL serialization, round-trip, and read semantics

33. **`upsert_turn` appends one complete UTF-8 JSON object per line.** Behavior: each append adds exactly one newline-terminated compact JSON object with all `TurnRecord` field names, including fields whose value is null. Why it matters: append-only crash recovery and schema stability depend on line granularity. Falsifiability: an implementation that pretty-prints multi-line JSON, omits null fields, or rewrites the file instead of appending fails. Required mock/fixture/setup: temp store, one complete turn with null optional fields, direct file inspection.

34. **Datetime fields serialize as RFC3339 and load back semantically equal.** Behavior: aware `created_at`, `ConversationRef.created_at`, and `updated_at` values serialize as RFC3339 strings and load back as equivalent datetimes; null remains null. Why it matters: stable JSONL/index schema and sorting use these timestamps. Falsifiability: an implementation that writes Python `repr(datetime)`, epoch floats, or local wall-clock conversions fails. Required mock/fixture/setup: fixed UTC datetime and null datetime records.

35. **Tuples/lists round-trip predictably.** Behavior: tuple fields such as `active_tools`, `attachments`, and `citations` serialize as JSON arrays and load back into the public immutable tuple/dataclass shapes. Why it matters: callers should not see mutable JSON internals after load. Falsifiability: an implementation that loads `active_tools` as a comma-separated string or raw dict list fails. Required mock/fixture/setup: turn with multiple tools, attachments, and citations.

36. **`load_transcript` returns a Transcript with ref, turns, and paths.** Behavior: loading by bare id, URL, project URL, alias, or `ConversationRef` returns a `Transcript` whose `conversation`, `turns`, `raw_mapping_path`, and `transcript_path` match the stored conversation. Why it matters: public history/export consumers need both metadata and durable paths. Falsifiability: an implementation that returns only a list of turns or loses project metadata fails. Required mock/fixture/setup: store containing one plain and one project conversation plus alias.

37. **Default reads hide all pending local stubs.** Behavior: `load_transcript(..., include_pending=False)` excludes records whose `message_id` starts `local:` and `turn_index is null`, regardless of whether they have been superseded. Why it matters: an unsubmitted prompt must never appear as real history/export output. Falsifiability: an implementation that hides only superseded stubs, or shows unsuperseded stubs by default, fails. Required mock/fixture/setup: transcript with an unsuperseded local stub and visible canonical turns.

38. **`include_pending=True` surfaces pending stubs without promoting them.** Behavior: explicit pending reads include local stubs while preserving their `turn_index=None`, `status="partial"`, and local id; they are still distinguishable from canonical turns. Why it matters: diagnostics/recovery need durable outbox visibility without confusing history. Falsifiability: an implementation that either still hides stubs or converts them into normal user turns with an index fails. Required mock/fixture/setup: same transcript as previous behavior loaded both ways.

39. **Canonical supersession hides the stub and keeps the real turn.** Behavior: after a canonical user record with `supersedes_message_id="local:<client_send_id>"` is appended, default reads show only the canonical user record; explicit pending reads can show both if the implementation chooses to surface all pending diagnostics. Why it matters: fixes the lose-nothing eager-write invariant without duplicating prompts. Falsifiability: an implementation that shows both by default, drops the canonical record, or deletes the stub physically fails. Required mock/fixture/setup: begin-send stub plus canonical user record with matching `client_send_id`/`supersedes_message_id`.

40. **Last-writer-wins by `message_id`.** Behavior: when multiple valid JSONL records share a `message_id`, `load_transcript` keeps the last valid record for that id. Why it matters: idempotent re-scrapes append corrected complete records rather than editing history. Falsifiability: an implementation that keeps the first record or concatenates duplicates fails. Required mock/fixture/setup: partial assistant record followed by complete replacement with same id and changed content/status.

41. **Loaded turns sort by the specified stable order.** Behavior: after filtering and last-writer-wins, turns are ordered by `(turn_index is null, turn_index, created_at or "", message_id)`. Why it matters: current-branch visible history must be deterministic and pending diagnostics should not disturb visible order. Falsifiability: an implementation that preserves append order after replacements or sorts lexicographically by `message_id` only fails. Required mock/fixture/setup: records appended out of order with turn indices 0/1/2, same timestamps, and a pending stub.

42. **Reader tolerates at most one torn trailing line.** Behavior: a transcript ending with one invalid partial JSON line loads prior valid records and emits a warning/diagnostic; invalid JSON in the middle or multiple invalid trailing lines fails with `StoreError` or an equivalent documented store read error. Why it matters: crash recovery must tolerate a torn append without silently masking real corruption. Falsifiability: an implementation that crashes on a single torn final line, or silently ignores a corrupt middle line, fails. Required mock/fixture/setup: hand-written JSONL files for single trailing tear, middle corrupt line, and two trailing corrupt lines.

43. **Append operations preserve prior complete lines after simulated failure.** Behavior: if an append fails after prior lines exist, already-fsynced complete records remain readable and no earlier content is truncated. Why it matters: transcript loss is one of the explicit gotchas. Falsifiability: an implementation that rewrites the whole file and leaves it empty/half-written on failure fails. Required mock/fixture/setup: temp transcript with one record and a failure injection in the file write/flush path for the next append.

44. **Concurrent append attempts do not interleave JSON objects.** Behavior: two store instances appending to the same conversation produce two valid complete lines in some order, never byte-interleaved JSON. Why it matters: the design calls for per-conversation advisory locking. Falsifiability: an implementation without locking that can produce `{"a"...{"b"...}` interleaving fails. Required mock/fixture/setup: temp store plus a controllable fake lock or two-process/thread fixture with small records; assert final JSONL parses line-by-line.

## G. Atomic replace/write helpers and attachment paths

45. **Raw mapping replace is atomic and validates before replacing.** Behavior: `write_raw_mapping_atomic(conversation_id, raw_tmp)` parses/validates the candidate raw mapping before `os.replace`; valid input replaces the old file, invalid input leaves the old raw mapping intact. Why it matters: raw mapping must be either old-complete or new-complete, never half-replaced. Falsifiability: an implementation that renames invalid JSON over the old file fails. Required mock/fixture/setup: existing `raw-mapping.json`, one valid tmp with top-level mapping/current_node, one invalid tmp.

46. **Raw mapping persistence does not store request headers.** Behavior: writing raw mapping persists backend JSON content only and rejects/strips any accidental request-header wrapper if the public contract receives one. Why it matters: M3 says store never receives auth/OAI headers and raw mapping must contain no request headers. Falsifiability: an implementation that writes `authorization`, `cookie`, or `oai-device-id` keys from a wrapper fixture into `raw-mapping.json` fails. Required mock/fixture/setup: candidate raw tmp with an obvious header wrapper/sensitive keys and expected failure or sanitized output, depending on chosen API semantics.

47. **Index replacement is atomic.** Behavior: if writing a patched index fails before rename, the previous complete `index.json` remains readable; a successful update is complete JSON, never a partial file. Why it matters: index is rebuildable, but corruption should not be self-inflicted. Falsifiability: an implementation that writes directly to `index.json` and leaves `{"schema` after failure fails. Required mock/fixture/setup: existing index and failure injection during temp write/rename.

48. **`atomic_write_payload` writes strings and bytes exactly.** Behavior: the helper accepts `str` and `bytes`, writes exactly the supplied payload to the requested path through a same-directory temp/rename, and creates parent directories when that is the documented behavior. Why it matters: `--out` must not corrupt markdown or binary-ish payloads. Falsifiability: an implementation that encodes bytes through `str(bytes)`, normalizes newlines, or truncates NUL-containing bytes fails. Required mock/fixture/setup: temp output paths with Unicode markdown and byte payload fixtures.

49. **Out-file atomicity preserves old payload on failure.** Behavior: if writing `--out` fails before rename, the previous output file remains unchanged and no final path points to partial content. Why it matters: stdout is fallback, but existing artifacts should not be destroyed by a crash. Falsifiability: an implementation that opens the final output path directly and truncates it before failing fails. Required mock/fixture/setup: preexisting output file and failure injection during write/fsync.

50. **`attachment_path` is confined to the conversation attachments directory.** Behavior: generated attachment paths are under `conversations/<id>/attachments/`, are safe against `../` or absolute filenames, and are deterministic for a given ref. Why it matters: attachment cache must not allow path traversal or scatter files. Falsifiability: an implementation that joins raw filename `../../secret` directly fails. Required mock/fixture/setup: attachment refs with normal, duplicate, absolute, and traversal-like filenames/sha values.

## H. Store send-stub and salvage persistence

51. **`begin_send` persists the conversation ref before the stub.** Behavior: `begin_send(ref, prompt, model, active_tools)` ensures the conversation/index entry exists and then appends the pending local user stub. Why it matters: a crash after prompt capture but before UI submission still leaves enough identity to recover. Falsifiability: an implementation that returns an in-memory stub without transcript/index durability fails. Required mock/fixture/setup: temp store, ref, prompt, then inspect index and transcript immediately after `begin_send`.

52. **`begin_send` creates a unique durable local stub with prompt content.** Behavior: each call uses a unique `client_send_id`, `message_id="local:<client_send_id>"`, role `user`, prompt in `content_markdown`, `turn_index=None`, `created_at=None`, `status="partial"`, `partial=True`, and preserves requested model/tools. Why it matters: this is the accepted pending eager-write design decision. Falsifiability: an implementation that reuses a constant id, omits prompt text, or marks stub complete fails. Required mock/fixture/setup: two `begin_send` calls with different prompts/tools.

53. **`commit_send` appends, not mutates, the canonical user record.** Behavior: committing a send appends the canonical user turn with `supersedes_message_id` pointing to the local stub and does not rewrite/delete the stub line. Why it matters: append-only recovery and auditability. Falsifiability: an implementation that edits the stub in place or removes it from JSONL fails despite default reads looking correct. Required mock/fixture/setup: begin-send stub, canonical user fixture, direct JSONL line count/content inspection.

54. **`record_partial` persists salvaged assistant output with redacted error.** Behavior: partial salvage creates/appends a `TurnRecord` with assistant role, supplied partial markdown, `status="partial"` or `"error"` per design choice but always `partial=True`, associated `client_send_id` when available, and `error` details that carry stable code/message without prompt/response/header secrets. Why it matters: long/failed completions must be recoverable without leaking credentials. Falsifiability: an implementation that only logs salvage, marks it complete, or stores raw exception repr containing bearer/cookie headers fails. Required mock/fixture/setup: fake `CompletionTimeoutError`/generic exception with sensitive details and partial markdown.

## I. Markdown rendering and stdout plus `--out` payload behavior

55. **Markdown render uses loaded visible turns only.** Behavior: `render_markdown(load_transcript(...))` renders exactly the default-visible canonical turns in stable order; pending local stubs and hidden/internal raw mapping data are absent unless an explicit pending/diagnostic render mode is added. Why it matters: `history`/`export` must not show unsubmitted prompts or hidden tool/system nodes. Falsifiability: an implementation that renders every JSONL line, including local stubs, fails. Required mock/fixture/setup: transcript with user/assistant canonical turns, local stub, replacement record, and raw mapping file containing hidden text sentinel.

56. **Markdown render preserves content markdown literally.** Behavior: rendered output includes each `content_markdown` body unchanged aside from deterministic role/turn separators; math tokens such as `\widehat`, `\ne`, `\frac{}{}` and code fences are not normalized/corrupted. Why it matters: faithful markdown is a core gotcha from the rewrite spec. Falsifiability: an implementation that renders via HTML/DOM text, escapes backslashes incorrectly, or strips code fences fails. Required mock/fixture/setup: turns containing LaTeX, markdown tables, code fences, and Unicode.

57. **Markdown render is deterministic.** Behavior: rendering the same `Transcript` twice produces byte-identical output and does not depend on current time, filesystem order, or Python object ids. Why it matters: `--out` artifacts and agent comparisons need stable outputs. Falsifiability: an implementation that adds `Rendered at <now>` or unordered attachment/citation dumps fails. Required mock/fixture/setup: fixed transcript rendered twice under different clock/env conditions.

58. **Render does not fetch attachments or citations.** Behavior: rendering may mention stored refs if the chosen format does so, but it never opens network connections or calls lazy attachment fetch. Why it matters: `history`/`export` are browser-free/local-only. Falsifiability: an implementation that tries to download a citation URL or materialize pending attachment bytes during render fails. Required mock/fixture/setup: transcript with pending attachment and citation URL plus fetch/open hooks that raise if called.

59. **Payload helper prints to stdout and writes `--out`; neither suppresses the other.** Behavior: the public output helper used by `ask`/`scrape` emits the payload to stdout every time and additionally writes the same payload to `--out` when provided. Why it matters: fixes the v1 `--out` suppresses stdout gotcha and is an M4 Step 2/6 acceptance point. Falsifiability: an implementation that chooses stdout OR file, or writes a summary to stdout and full payload to file, fails. Required mock/fixture/setup: capsys-like stdout capture, temp `--out` path, known markdown payload.

60. **Payload helper attempts stdout and out-file independently.** Behavior: if out-file writing fails, stdout still contains the payload; if stdout capture/write fails in a controlled fixture, out-file success/failure is reported independently according to the public error contract. Why it matters: truncation/session-loss gotcha demands stdout as fallback and durable store as truth. Falsifiability: an implementation that writes `--out` first and returns before printing stdout on file error fails. Required mock/fixture/setup: failing output path or monkeypatched payload writer plus stdout capture.

61. **Payload helper does not preflight CDP.** Behavior: rendering/printing a stored payload uses store/local data only and never constructs a CDP channel or calls `preflight`. Why it matters: history/export and local rendering must be offline. Falsifiability: an implementation that calls session preflight from a shared CLI setup path before writing local history fails. Required mock/fixture/setup: fake channel/preflight object that raises if touched and a local transcript.

## J. Browser-free history/export over the store

62. **`history` reads local JSONL only.** Behavior: public `Session.history` and/or CLI `history <conv>` resolves the conversation, loads the local transcript, renders markdown, and returns/prints it without CDP preflight, attach, open-tab, or network. Why it matters: M4 Step 2 acceptance explicitly says history/export no CDP preflight. Falsifiability: an implementation that preflights CDP in `Session.__enter__` or CLI global setup before history fails. Required mock/fixture/setup: local store with transcript; monkeypatched `CdpChannel.preflight`, `BrowserChannel.attach`, and network calls that raise if invoked.

63. **`export` reads local JSONL only and supports `--out` without suppressing stdout.** Behavior: public `Session.export` and/or CLI `export <conv> --out file` uses the same local render and payload helper as history; it does not preflight CDP and writes both stdout and file. Why it matters: export is a local rendering verb and inherits the stdout+out acceptance point. Falsifiability: an implementation that treats export like scrape and opens a browser, or suppresses stdout when `--out` is passed, fails. Required mock/fixture/setup: local transcript, out path, preflight-raising fake, stdout capture.

64. **History/export default visibility matches `Store.load_transcript`.** Behavior: default history/export output excludes pending local stubs; an explicit `--include-pending`-style option, if exposed in M4, includes them as diagnostics without pretending they are canonical turns. Why it matters: unsubmitted prompts must not appear in normal history/export. Falsifiability: an implementation that calls `load_transcript(include_pending=True)` by default fails. Required mock/fixture/setup: transcript with canonical turns plus unsuperseded stub, invoked through store/session/CLI public path.

65. **History/export resolve URL shapes and aliases consistently.** Behavior: local history/export accepts bare id, plain `/c/<id>` URL, project `/g/g-p-<projid>/c/<chatid>` URL, and aliases that resolve to the same stored conversation. Why it matters: identity parsing and store addressing must compose. Falsifiability: an implementation that can render by bare id but fails the project URL or requires index for a URL fails. Required mock/fixture/setup: one stored plain conversation, one stored project conversation, and aliases/sessions in index.

## Design ambiguities to resolve or pin before writing tests

- Project id normalization: M3 alternates between `/g/g-p-<project_id>/...` and lead shorthand `/g/g-p-<projid>/...`; tests must pin whether `ConversationRef.project_id` stores `projid` or `g-p-projid`, then require URL round-trip consistency.
- Markdown render exact format is not specified beyond faithful visible-turn content. Tests should first define a minimal deterministic format (role labels/separators and optional citation/attachment reference formatting) and avoid overfitting beyond that.
- Selector validation entry-point/name is not specified. Tests should target the chosen public loader/validator, not private JSON parsing details.
- Data-model validation location is not specified. Tests should assert public behavior (invalid records cannot be durably persisted/loaded as valid), whether enforcement lives in dataclass `__post_init__`, store validation, or constructors.
- `include_pending=True` wording clearly surfaces pending stubs, but exact rendering of superseded stubs in diagnostics is not fully specified. The checklist assumes default hides all stubs and explicit pending reads may show all local stubs while keeping them marked partial.
- Torn-line “warning” mechanism is unspecified. Tests should require an observable warning/diagnostic through the project’s selected logging/warnings interface once chosen.
- Allowlist scheme policy and `sanitize_for_log` exact redaction are not fully spelled out in M3; the safe default in tests should reject non-http(s) navigation/fetch targets and redact credentials/query/fragment while retaining host context.
- Naive datetime handling is unspecified. Prefer requiring aware datetimes/RFC3339 UTC-equivalent serialization at the public boundary rather than silently interpreting local time.
- Index rebuild semantics after corruption are under-specified. Tests should require stateless URL/id reads to survive corrupt index, while alias recovery can be best-effort unless implementation documents a stronger rebuild.
